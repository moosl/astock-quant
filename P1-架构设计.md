# P1 — 架构设计

> architect 产出 · 2026-05-15 · 基于 P0 选型报告 + 实际研读 4 个参考 repo
>
> 前置：`P0-框架选型.md`、`zazzy-wondering-tiger.md`（完整计划）。本文档定方案 A/B、给出项目骨架蓝图。

---

## 0. 一句话结论

**选方案 A 的「瘦身版」**：复用成熟代码（v1 回测器逻辑、`a_stock.py` 数据封装、两个 repo 的因子计算片段）作为**实现参考**，但 Stage 1 **不引入任何重框架依赖**（无 LangGraph / langchain / backtrader）。项目自建一套干净骨架，目录分层照搬 ai-hedge-fund v2 的设计哲学（`Data → Factors → Models → Backtest → Signals`），数据契约用 Pydantic，4 类预测目标共用一套基础设施、Stage 1 只把 ① 涨跌方向二分类做透。

骨架核心结构（已 scaffold 在 `量化/` 根目录）：

```
astock_quant/
  config/      配置（universe、路径、回测参数）
  data/        数据层 —— 抽象 Protocol + a_stock 适配器 + 数据集构建
  factors/     因子层 —— 量价因子 + 财务因子 + 资金流因子 +【LLM 因子预留】
  labels/      标签层 —— 把「未来 N 日涨跌/收益/排序」打成训练 target
  models/      模型层 —— 4 类预测目标，Stage 1 做透二分类
  backtest/    回测层 —— 逐日回测引擎 + A股约束 + 绩效指标
  signals/     信号层 —— 模型预测 → 可读买卖/持仓信号
  pipeline/    端到端编排 —— 一条命令跑通 ①
contracts.py   全局数据契约（Pydantic models，模块间传这个）
```

## 1. 方案 A vs 方案 B：决策与理由

P0 报告把这个决策留给 architect。我的选择和理由如下。

### 1.1 决策：方案 A（瘦身版）

不是「原始方案 A」（直接 `pip install` 框架、连 LangGraph），也不是「纯方案 B」（完全不看框架代码、全凭空写）。而是两者的中间形态，我称为**方案 A 瘦身版**：

| 维度 | 做法 |
|---|---|
| **框架作为 pip 依赖** | ❌ 不引入。`pyproject.toml` 里没有 langchain / langgraph / backtrader。Stage 1 的直接依赖只有 `pandas / numpy / lightgbm / scikit-learn / mootdx / akshare / requests / stockstats / pydantic / matplotlib`。 |
| **成熟代码作为实现参考** | ✅ 重度复用。`backtest/` 的逐日推进逻辑参考 ai-hedge-fund v1 `src/backtesting/`；`data/` 的 mootdx 封装 + 缓存 + look-ahead 防护参考 `a_stock.py` 的 `_load_ohlcv_astock`；`factors/` 的指标计算参考两个 repo 的 `technicals.py` / `get_indicators`。**研读后用自己的话重写进我们的骨架**，不直接 import 框架包。 |
| **项目骨架** | ✅ 自建。目录分层 + 数据契约照搬 ai-hedge-fund **v2** 的设计哲学（v2 README 的 `Data→Signals→Features→...` 分层、`BaseSignal` ABC、Pydantic 契约、`Protocol` 数据抽象）。 |
| **`.p0-repos/`** | 保留在原地，作为「实现时随时翻」的参考库，不打包进项目、不进 import path。 |

### 1.2 为什么不是纯方案 A（连框架依赖）

- 3 个候选框架的核心（LangGraph 多 Agent 辩论拓扑）和我们的混合架构**正交** —— 我们要「LLM 出因子 → 喂 ML 模型」，不要「AI 辩论出决策」。引入 LangGraph 只是给项目堆概念负担。
- `backtrader` 在 TradingAgents-astock 里本身就是**摆设依赖**（声明了但全仓库零 `import`）—— 没有复用价值。
- ai-hedge-fund v1 回测器虽然完整，但**绑定美股付费数据 API**，直接 import 跑不起来；它的价值是「逐日回测的代码逻辑」，研读后重写比硬接更干净。
- 学习型项目，重依赖 = 调试时多一层不可控的黑盒。

### 1.3 为什么不是纯方案 B（完全不看框架）

- ai-hedge-fund **v1 回测器**是整个选型里唯一完整可跑的回测引擎，它的逐日推进 / 持仓管理 / Sharpe-Sortino-回撤计算是经过验证的，从零硬写容易出错（尤其 look-ahead bias、T+1 约束）。研读它的逻辑能少踩坑。
- ai-hedge-fund **v2 的设计**（`BaseSignal` ABC、Pydantic 数据契约、`Protocol` 数据抽象）是「干净量化项目长什么样」的现成最佳范本，照抄设计哲学能让骨架一开始就站对地方。
- `a_stock.py` 的 `_load_ohlcv_astock` 已经把「mootdx 拉数 → CSV 缓存 → 按日期截断防未来函数」这个易错环节写对了，研读后重写比凭空写稳。
- 「完全不看、全凭空写」对学习型项目没有额外好处，只会重复造轮子 + 重新踩别人踩过的坑。

### 1.4 方案 A 瘦身版的代价与应对

- **代价**：`backtest/` 和 `data/` 的核心代码要「研读参考 → 重写」，比直接 `import` 多一点工作量（落在 P2 data-engineer、P4 model-engineer 头上）。
- **应对**：这点工作量换来的是「项目里每一行都是自己的、可读可改可调试」—— 完全符合学习型项目的定位。而且参考代码就在 `.p0-repos/`，重写不是凭空写，是「看着标准答案理解后誊写」。

## 3. 各模块职责

### 3.1 `contracts.py` —— 全局数据契约（核心）

> 思路来自 ai-hedge-fund v2 `models.py`：用 Pydantic 定义「模块之间传什么」，单一事实来源，跨模块不传裸 dict / 裸 DataFrame（DataFrame 仅在模块**内部**用，跨模块边界用 contract 包一层或约定 schema）。

Stage 1 需要的契约：

| 契约 | 用途 | 关键字段 |
|---|---|---|
| `PriceBar` | 单根 OHLCV K线 | ticker, date, open/high/low/close, volume, amount |
| `FinancialMetrics` | 单只股票某报告期财务指标 | ticker, report_period, eps, roe, revenue, net_profit, pe, pb... |
| `MoneyFlowRecord` | 单只股票某日资金流 | ticker, date, northbound, main_inflow, dragon_tiger... |
| `NewsItem` | 单条新闻/研报/公告文本（Stage 2 用，Stage 1 先定义） | ticker, date, title, content, source |
| `FactorValue` | 单个因子在 单ticker×单日 的值 | factor_name, ticker, date, value, （可选 z_score / percentile） |
| `FactorFrame` | 一批因子值的集合（对应 v2 的 QuantSignals 思路） | 内部 DataFrame：index=(date,ticker), columns=因子名 |
| `Label` | 单个训练标签 | ticker, date, target_type, value（①是0/1，②是float，③是rank） |
| `Prediction` | 模型单条预测输出 | ticker, date, target_type, value, score, （可选 proba） |
| `BacktestResult` | 回测结果 | equity_curve, metrics（dict）, trades, positions |
| `SignalReport` | 信号层产出 | date, signals（ticker→buy/hold/sell + 强度/理由） |

`target_type` 用 `Literal["direction", "return", "ranking", "trade_signal"]` —— **这是 4 类预测目标共用基础设施的关键锚点**：同一个 `Label` / `Prediction` 契约，靠 `target_type` 区分，下游 `models/` 和 `signals/` 按 type 分派。

### 3.2 `config/` —— 配置

`settings.py`：股票池定义（Stage 1 可以先用一个小 universe，如沪深300 成分或手选 20-50 只）、数据缓存路径、回测起止日期 / 初始资金 / 手续费率、因子计算窗口、训练/验证切分参数。集中一处，避免散落魔法数字。

### 3.3 `data/` —— 数据层

| 文件 | 职责 |
|---|---|
| `protocol.py` | `DataSource` Protocol —— 定义数据源必须实现的方法（`get_prices` / `get_financials` / `get_moneyflow` / `get_news`），结构化类型，任何实现这些方法的类都能当数据源。思路来自 v2 `data/protocol.py`。**这是「换数据源」和「Stage 2 接文本源」的扩展位。** |
| `astock_source.py` | A股数据适配器，实现 `DataSource` Protocol。研读 `a_stock.py`（mootdx / 腾讯财经 / akshare 封装）后重写精简版，只留我们要的端点。返回 `contracts.py` 的 Pydantic 对象。 |
| `cache.py` | CSV 缓存 + **look-ahead 截断**（按 `curr_date` 截断历史，防未来函数）。逻辑参考 `_load_ohlcv_astock`。 |
| `dataset.py` | **P2 主要工作量** —— 把单股票查询扩展成「多股票 × 长时间」的 panel 数据集：universe 循环、落地、增量更新。P0 报告明确指出 a-stock-data 没有现成的批量 universe 能力，这里要自己写。 |

### 3.4 `factors/` —— 因子层（含 LLM 因子预留）

| 文件 | 职责 |
|---|---|
| `base.py` | `BaseFactor` ABC —— 所有因子的统一接口（`name` 属性 + `compute(panel_data) -> FactorValue/FactorFrame`）。思路来自 v2 `signals/base.py` 的 `BaseSignal`。 |
| `price_volume.py` | 量价因子：动量、均值回归、波动率、换手率、技术指标（RSI/MACD/BOLL/ATR）。计算逻辑参考 ai-hedge-fund v1 `technicals.py` + `a_stock.py` 的 `get_indicators`（stockstats）。 |
| `fundamental.py` | 财务因子：估值（PE/PB）、盈利（ROE/净利率）、成长（营收/利润增速）。数据来自季报。 |
| `moneyflow.py` | 资金流因子：北向资金、主力净流入、龙虎榜。注意 P0 报告提醒部分端点只有短历史，长周期回测要评估。 |
| `llm_factor.py` | **【Stage 2 预留】** LLM 情绪/事件因子。Stage 1 **只放 stub**：定义 `LLMFactor(BaseFactor)` 的类骨架 + docstring 说明「Stage 2 在此接入：文本源 → LLM 抽取 → 结构化因子值」。因为它继承 `BaseFactor`、产出同样的 `FactorValue`，Stage 2 接入时**对 `registry` 和下游模型完全透明**。详见第 5 节。 |
| `registry.py` | 因子注册表 + 批量计算编排：把启用的因子跑一遍，产出对齐的 `FactorFrame`。新增因子（含 LLM 因子）只需注册，不改下游。 |

### 3.5 `labels/` —— 标签层

`targets.py`：把「未来 N 日的表现」打成 ML 训练标签。**4 类预测目标的标签都在这里生成**，靠 `target_type` 区分：

- `direction`（①做透）：未来 N 日收益 > 阈值 → 1，否则 0（二分类）
- `return`（②扩展点）：未来 N 日收益率 / 价格（回归 target）
- `ranking`（③扩展点）：横截面上未来 N 日收益的排序 / 分位
- `trade_signal`（④扩展点）：基于未来路径的买卖点标注

Stage 1 把 `direction` 写完整，②③④ 留函数 stub + docstring。**关键**：4 类标签共用同一套 `data → factors` 上游，只是 `labels` 这一步和 `models` 这一步按 type 分派 —— 这就是「共用基础设施」的体现。

### 3.6 `models/` —— 模型层（4 类预测目标）

| 文件 | 职责 |
|---|---|
| `base.py` | `BasePredictor` ABC —— `fit(X, y)` / `predict(X) -> Prediction` / `save` / `load` 统一接口。4 类目标的模型都实现它，下游回测/信号层面对统一接口。 |
| `direction.py` | **① 涨跌方向二分类（Stage 1 做透）** —— LightGBM 二分类，完整实现：特征对齐、训练、预测、概率输出、特征重要性。 |
| `ret_regression.py` | ② 收益率/价格回归 —— **扩展点 stub**：类骨架 + docstring，标明「复用 BasePredictor，换 LightGBM regressor + 回归 label」。 |
| `ranking.py` | ③ 选股排序 横截面 Top N —— **扩展点 stub**：标明「复用 BasePredictor，可用 LightGBM ranker 或回归分数排序」。 |
| `trade_signal.py` | ④ 买卖信号 —— **扩展点 stub**：标明「复用 BasePredictor + signals 层」。 |
| `splits.py` | 时序安全的数据切分 —— 训练/验证按时间先后切，**带 purge gap**（防止标签的未来窗口泄漏到训练集）。这是 ML 量化最容易错的地方，独立成文件。 |

LSTM 暂不在骨架里铺文件 —— Stage 1 树模型为主（P0 决策：算力非瓶颈走树模型）。LSTM 若 P3 要做，作为 `direction.py` 同级新增 `direction_lstm.py`，同样实现 `BasePredictor`，骨架已为它留好位置。

### 3.7 `backtest/` —— 回测层

| 文件 | 职责 |
|---|---|
| `engine.py` | 逐日回测引擎 —— 逐日推进、按预测/信号下单、更新持仓、记录净值。**逻辑重度参考 ai-hedge-fund v1 `src/backtesting/engine.py` + `controller.py`**，研读后重写。 |
| `portfolio.py` | 持仓 + 现金 + 交易成本管理。参考 v1 `portfolio.py`。 |
| `constraints.py` | **A股专属约束**：T+1（买入次日才能卖）、涨跌停（停板不能成交）、最小手数（100 股）、ST 股处理。这是 v1 美股回测器没有的，要自己加 —— 参考 TradingAgents-astock 对 A股 交易规则的处理。 |
| `metrics.py` | 绩效指标：累计收益、年化、Sharpe、Sortino、最大回撤、胜率、对基准（沪深300）。参考 v1 `metrics.py` + `benchmarks.py`。 |

### 3.8 `signals/` —— 信号层

`generator.py`：把模型 `Prediction` 转成人能读的买卖/持仓信号（`SignalReport`）。4 类预测目标共用：`direction` → 涨/跌信号；`return` → 按预期收益排序；`ranking` → Top N 持仓清单；`trade_signal` → 直接买卖点。按 `target_type` 分派。

### 3.9 `pipeline/` —— 端到端编排

`run_direction.py`：把 ① 涨跌方向 的完整链路串成一个函数 —— `数据拉取 → 因子计算 → 标签生成 → 切分 → 训练 → 回测 → 信号产出`。对应计划里 Stage 1 验收的「全流程一条命令可复现」。②③④ 做的时候各加一个 `run_*.py`，复用同样的编排骨架。

### 3.10 `scripts/` —— 入口

`run_pipeline.py`：命令行薄壳，解析参数后调 `pipeline/`。`uv run python scripts/run_pipeline.py` 即可跑通 ①。

## 4. 4 类预测目标如何共用基础设施

这是本架构的核心设计点。**① 涨跌方向是一个完整的纵向切片，②③④ 是接在同一套基础设施上的扩展点。**

### 4.1 共用的部分（4 类目标完全复用，不改一行）

```
config/  →  data/  →  factors/        ← 这三层 4 类目标 100% 共用
（股票池）  （拉数）  （量价/财务/资金流/LLM因子）
```

无论预测「涨跌」「收益」「排序」还是「买卖点」，**输入特征 X 是同一套** —— 都来自 `factors/` 算出的 `FactorFrame`。数据层、因子层一次建好，4 类目标共享。

### 4.2 按 `target_type` 分派的部分（扩展点）

```
labels/targets.py   ── 按 target_type 生成不同 label
models/             ── 按 target_type 用不同 predictor（都实现 BasePredictor）
signals/generator   ── 按 target_type 转不同信号
```

这三层的代码靠 `target_type: Literal["direction","return","ranking","trade_signal"]` 分派。**扩展点的「接入位」非常明确**：

| 要加的目标 | 改哪里 | 改什么 | 不用动什么 |
|---|---|---|---|
| ② 收益率回归 | `labels/targets.py` + `models/ret_regression.py` | 加回归 label 函数 + LightGBM regressor 实现 | data / factors / backtest engine / signals 框架 |
| ③ 选股排序 | `labels/targets.py` + `models/ranking.py` | 加横截面排序 label + ranker 实现 | 同上 |
| ④ 买卖信号 | `labels/targets.py` + `models/trade_signal.py` + `signals/generator.py` 补分支 | 加买卖点 label + 模型 + 信号转换分支 | data / factors / backtest engine |

### 4.3 Stage 1 的纵向切片（① 做透）

Stage 1 把这一条**整条链路打通、跑出真实结果**：

```
config（小universe）→ data（拉真实A股数据）→ factors（量价因子先做，财务/资金流可后补）
→ labels.direction（未来N日涨跌二分类标签）→ splits（时序切分+purge）
→ models.direction（LightGBM 二分类，训练+预测+概率）
→ backtest.engine（逐日回测+A股约束+指标）→ signals.generator（涨跌信号）
→ pipeline.run_direction（一条命令串起来）
```

②③④ 在各自文件里是 **stub（类骨架 + docstring 说明接入方式）**，不写实现 —— 实现是 P3+ 视进度推进的事。但因为基础设施和接入位已经设计好，补 ②③④ 时**不需要重构**，只在上表指定的少数文件里加代码。

## 5. Stage 2 的 LLM 因子接口预留

P0 报告明确：LLM 的角色是「把新闻/研报/公告转成情绪/事件因子，作为一路因子喂给量化模型」，不直接出预测。架构必须让 Stage 2 能**平滑接入**，不动 Stage 1 已建好的东西。

### 5.1 预留的三个接入位

**接入位 1 —— 数据层 `data/protocol.py` 的 `get_news`**
`DataSource` Protocol 里 Stage 1 就定义好 `get_news(ticker, start, end) -> list[NewsItem]` 方法签名，`NewsItem` 契约 Stage 1 就在 `contracts.py` 里定义。`astock_source.py` 可以 Stage 1 先实现（`a_stock.py` 本来就有 `get_news` / `get_global_news`），也可以 Stage 2 再补 —— 但**接口形状 Stage 1 就固定**。

**接入位 2 —— 因子层 `factors/llm_factor.py`**
Stage 1 放一个 `LLMFactor(BaseFactor)` 的 **stub**：
- 它继承 `BaseFactor`，和量价因子是**同一个父类、同一个接口**
- 它的 `compute()` 产出的是同样的 `FactorValue` —— 只是 value 来自「文本 → LLM → 结构化抽取」而不是「数学公式」
- docstring 写清楚 Stage 2 的实现路径：`get_news 拿文本 → LLM 调用（参考 TradingAgents-astock llm_clients/）→ 结构化输出情绪分/事件标记（参考其 structured.py）→ 包成 FactorValue`

**接入位 3 —— 因子注册表 `factors/registry.py`**
`registry` 是「把启用的因子跑一遍产出 `FactorFrame`」。Stage 2 的 LLM 因子**只需在 registry 里注册一行**，就自动进入 `FactorFrame` 的列、自动成为模型的输入特征。

### 5.2 为什么这样能「平滑接入」

关键在于 **LLM 因子和量价因子在架构里是平级的** —— 都是 `BaseFactor` 的子类，都产出 `FactorValue`，都经 `registry` 汇成 `FactorFrame`。对下游（`labels` / `models` / `backtest` / `signals`）来说，**多了一路 LLM 因子和多了一个量价因子没有任何区别** —— 下游只看到 `FactorFrame` 多了几列。

所以 Stage 2 接 LLM 因子，**只动 `factors/llm_factor.py`（填实现）+ `factors/registry.py`（注册一行）+ 可能补 `data/astock_source.py` 的 `get_news`**，Stage 1 的数据集构建、模型、回测、信号**一行都不用改**。这正是计划里「Stage 2 平滑接入」的要求。

P0 报告指出可复用的 Stage 2 资产（届时从 `.p0-repos/TradingAgents-astock/` 取）：`llm_clients/`（多供应商 LLM 封装，含国产）、`agents/utils/structured.py`（LLM 结构化输出）、7 个分析师 prompt（政策/游资/解禁/舆情，A股语境中文）。
