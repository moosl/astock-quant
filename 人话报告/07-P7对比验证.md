## 四、对照实验结果：bit-identical（一字不差）

这次跑了 30 只票 × 4 年数据 × 两组（A 关 LLM / B 开 LLM）。两组成绩单：

| 指标 | A 组（关 LLM，25 因子）| B 组（开 LLM，26 因子）| 差距（delta）|
|---|---:|---:|---:|
| 因子数量 | 25 | 26（多了 news_sentiment）| +1 |
| 训练集行数 | 25,283 | 25,283 | 0 |
| 验证集行数 | 5,780 | 5,780 | 0 |
| **AUC** | **0.5131** | **0.5131** | **0.0**（一字不差）|
| **Accuracy** | **0.5334** | **0.5334** | **0.0** |
| **Log loss** | **0.6906** | **0.6906** | **0.0** |
| news_sentiment 重要性 | — | **0.0** | — |

### 4.1 怎么读这个表

> **名词解释·AUC**（报告 03 讲过）：评估模型好不好的常用指标，0.5 = 抛硬币、1.0 = 神模型。我们 Stage 1 是 0.5131（接近随机）。

两组 AUC **小数点后 4 位**都完全相同。auditor 实测连**小数点后 16 位**也完全相同（0.5131337782587783）。这就是「**bit-identical**」——一个 bit 都不差。

加 LLM 因子前后 AUC、Accuracy、Log loss 三个数字**全部不变**。意思是——

> **模型加了 LLM 因子之后，预测能力跟没加完全一样。**

### 4.2 "重要性 = 0" 是什么意思

LightGBM 这个工具有个特别有用的功能——它能告诉你「我训练的时候，每个因子被我用了多少次」（叫 feature importance / 重要性）。报告 03 看过 Top 15 因子重要性表，量价类的 `volatility_20d` 是 296 分最高。

这次 LLM 因子的重要性是 **0.0**。打个比方：

> **类比：模型像招了个能听懂中文的实习生，结果连他工位旁边走过一次都没去**。
>
> 26 个因子摆在 LightGBM 面前，让它自由选用。它用了 25 个老的量价/财务/资金流因子（每个都被用了几百次），但 LLM 这个新因子——**它一次都没用过**。不是它不知道有这个因子（n_features 显示 26 列），是它**主动选择不用**。

为什么？因为这个因子里 99% 都是空值（NaN）——下面这一节就讲为什么。

## 五、为什么 LLM 因子没用——是数据问题，不是 bug

这是 P7 这份报告**最重要的一节**。"模型没用 LLM 因子"听起来像 bug，但 auditor 反复审核了 6 项 + 跑了 178 次真实 API 调用确认——**因子代码本身完全正确，DeepSeek API 工作正常，数据完全走通**。问题出在更上游——**新闻数据本身太稀**。

### 5.1 因果链（4 步，每步有具体数字）

```
第 1 步：akshare（我们用的新闻数据源）有个硬限制
        ─→ 每只股票，每次最多只能拉到最近 ~20 条新闻

第 2 步：算一下总数
        ─→ 30 只股票 × 20 条新闻 = 总共 ~600 条新闻
        ─→ 而且全部集中在最近 6 周（2026 年 4 月 ~ 5 月）

第 3 步：但训练用的是 2022~2026 共 4 年数据
        ─→ 4 年大约有 1000 个交易日 × 30 只票 ≈ 30000 个"日期-股票"位置
        ─→ 600 条新闻 ÷ 30000 个位置 = 覆盖率不到 3%
        ─→ 训练集 25,283 行里，LLM 因子有实际数值的不到 1%
        ─→ 99%+ 的行是 NaN（空值）

第 4 步：LightGBM 在 99% 都是空值的列上自然打不出 importance
        ─→ importance = 0 是正确行为，不是 bug
```

### 5.2 用「上学」类比讲透

> **类比**：你培养孩子 4 年。LLM 老师只在毕业前最后 1 周来上课。问期末考试有没有提升？没法证明——他只来了那么短时间，孩子还没学到东西就考试了。
>
> 这不是说 LLM 老师水平差，是**他根本没机会教**。

或者换个角度：

> **类比**：你雇了那个 7×24 小时不睡觉的分析师（报告 06 讲过的）。结果他来上班第一天，发现他要分析的 30 只股票里——**每只只有最近 6 周的新闻能拿到，前 4 年的新闻全部联系不到**。所以他能给的分数也只覆盖最近 6 周。模型在前面 4 年的训练里看到的 LLM 列全是空白，自然不会学到「这个因子有用」。

### 5.3 关键的工程发现（auditor 强调这一点）

这次的"importance = 0"和**v1 的 bit-identical 是两回事**（下一节专门讲）。这次：

- **wiring（接线）通了**——5 处接线 bug 全修了，全链路 grep 验证 + 2 个守护测试盯着
- **DeepSeek API 工作正常**——178 次真实调用，全部成功返回 JSON 情绪分
- **缓存正常工作**——一条新闻一辈子只调一次 LLM
- **因子代码正确**——单条新闻分数、加权聚合、索引对齐全对

**唯一的问题在数据源**。akshare 的 `stock_news_em` API 有"最近 ~20 条"硬限制，不是我们改一行代码能解决的。

## 六、v1 vs v2：两次 bit-identical 但根因完全不同（重要）

P7 跑了两次。第一次（v1，上午跑的）出来也是 AUC bit-identical（一字不差）；第二次（v2，下午跑的）也是 bit-identical。**听起来一样，但根因完全不同**——这件事必须给你讲清楚，**因为它涉及到"我们到底有没有真的修好"**。

### 6.1 v1：上午那次，是 bug（代码问题）

v1 跑完后，verifier 发现：「咦？AUC 完全没变，DeepSeek 也一次没被调用——是不是哪里 bug 了？」

排查后发现 —— **真的有 bug**。报告 06 在墙上"插了 AI 设备"，但忘了**接最后一根电源线**。具体是：

```
run_direction (主入口)
   ↓ ❌ 这里没把 news_fetcher 传下去
compute_factor_frame (算因子总函数)
   ↓ ❌ 这里也没接收
fac.compute() (LLM 因子的算分方法)
   ↓ news_fetcher 永远是 None
   ↓ → 拿不到新闻 → 全 NaN → LLM 一次没被调用
```

**5 处接线缺失，导致整条链路 LLM 因子根本没机会工作**。这是个**纯 bug**——AI 设备插好了、但电源线断了。

### 6.2 v1 到 v2 之间，factor-engineer 修了 5 处接线 bug

5 处接线（auditor 已审 PASS）：

1. **dataset 返回 source**（数据源加进结果字典）
2. **registry 加 news_fetcher 透传参数**（算因子总函数能传下去新闻拉取器）
3. **pipeline 注入 source.get_news**（主入口真把"新闻拉取器"传进去）
4. **llm_factor 从 kwargs 取 fetcher**（LLM 因子代码真用上传进来的拉取器）
5. **max_tokens 256 → 512**（顺便修了 v1 Step 1 暴露的"AI 输出被截断"小问题）

修完后加了 **2 个守护测试**：
- 一个测试**直接断言 wiring 必须通**：fetcher 被调用 ≥ 1 次 + 2 只票都被拉到新闻 + LLM 因子有 ≥ 3 个非 NaN 值
- 测试 docstring 写明：「**如果未来回退把 news_fetcher 透传删掉，P7 bit-identical 会立刻在 CI 复现**」——把根因预防写进测试

### 6.3 v2：下午那次，wiring 真通了，跑出来还是 bit-identical

修完之后重跑 v2。这次：

- **DeepSeek 真的被调用 178 次**（不是 0 次）
- **真的烧了 ¥0.09**（不是 ¥0）
- **缓存里真的多了 30 个新闻情绪 JSON 文件**

但 **AUC 还是 bit-identical**。这次的"一字不差"**不是 bug，是数据稀的客观结果**——99% 的训练样本 LLM 因子是 NaN，LightGBM 在那些行上根本拿不到信号。

### 6.4 v1 vs v2 对比表

| 维度 | v1（上午）| v2（下午）|
|---|---|---|
| wiring 状态 | ❌ 未接线（bug）| ✅ 5 处修复，全通 |
| 实际 LLM 调用 | **0** 次（设备插好但电源线断了）| **178** 次（真在工作）|
| 真烧的钱 | ¥0 | ¥0.09 |
| LLM 因子 NaN 比例 | 100%（完全没数据）| 99%（有一点点数据）|
| AUC delta | 0.0 bit-identical | 0.0 bit-identical |
| 根因 | **代码 bug**（接线缺失）| **数据源限制**（akshare 只返最近 20 条）|

### 6.5 这次区分讲清的工程意义

**两次都失败，但失败原因完全不同，处理方式也完全不同**：

- v1 那种 bug 是**能修的**——补 5 行代码 + 加 2 个测试，搞定
- v2 那种数据问题**改不了**——除非换新闻源 / 换数据策略，否则不管再跑多少次都还是 bit-identical

auditor 在审核里特别强调这一点的诚信价值：**承认两次都失败，但分清失败类型**——这是诚信汇报里最难的事。

### 7.2 P7 的真正成就：诚信地否决了一个假设

诚实地说「**当前数据源下 LLM 无用，瓶颈在数据**」——下一步就清楚了：**别花时间调 prompt / 换模型，先解决新闻覆盖问题**。

## 八、下一步——3 条路，你选了第 A 条

P7 跑完，技术文档列了 3 条可选的下一步。lead 把你的选择标了一下（你选了 **A**）——但 3 条路都给你解释一下，让你知道**为什么 A 是合理的选择**。

### 8.1 路 A：接受现状，进 P8 收尾 Stage 2（你选了这个）✅

**说法**：「我们用 ¥0.09 + 半小时算力，买到了一个真知识——当前 akshare 数据源下 LLM 因子无用，瓶颈不在算法在数据。**这个负面发现本身就是 P7 的成果**。直接进 P8 收尾 Stage 2。」

**为什么这是合理选择**：

- Stage 2 验收条件（②③④）实际上已经达成 —— 加 LLM 因子前后回测对比已经做完，"对比"出来是 0，但**对比本身完整跑完了**
- 继续投入更多时间换数据源、调 prompt、扩窗口——**回报是不确定的**，可能再花一周还是 importance=0
- 这是"学习型项目"，**核心目的本来就不是"出 alpha"**（报告 05 一直强调这点），是"跑通 + 守纪律"
- 留下完整的 P7 文档，未来你或者别人接手时——「想做"扩新闻源"的话从哪开始」一目了然

**结论**：**用 ¥0.09 学到「LLM 因子在当前数据条件下无用」这个真知识，比花一周追逐不确定的 alpha 更值**。

### 8.2 路 B：换数据源 / 加新闻覆盖（你没选）

**做法**：换掉 akshare 这个新闻源，找能拉历史新闻的：
- 同花顺 / 东方财富研报 API
- Choice / 万得这种付费数据源
- akshare 加翻页（如果支持的话）
- 或者用**公告替代新闻**——公告（财报、重大事项公告）历史覆盖比新闻好

**代价**：要做数据源调研、可能要付费、要重新写数据接入。投入至少 1 ~ 2 周。

### 8.3 路 C：短窗口大 universe（你没选）

**做法**：只在最近 6 周（有新闻覆盖的那段）跑对照实验——绕开历史数据稀疏的问题。

**代价**：6 周数据量太小（不到 1000 行训练样本），结论不可靠。即便看到 AUC 上升，也可能是噪音，不是真信号。

### 8.4 三条路对比

| 路 | 投入 | 预期回报 | 风险 |
|---|---|---|---|
| **A**（接受现状进 P8）| 极低 | 明确的 negative finding 已落袋 | 0 |
| B（换数据源）| 1~2 周 | 可能终于看到 LLM 因子有效 | 也可能换了还是 0 |
| C（短窗口实验）| 几小时 | 可能看到小幅 AUC 变化 | 样本太小，结论不可靠 |

## 十一、研究的本质：一个负面发现，跟一个正面发现同样有价值

最后说一段稍微 reflective 的话。

科学/研究里有句话：「**一个被严谨实验否决的假设，跟一个被验证的假设同样有价值**」。

如果 P7 跑出来 AUC 从 0.5131 上升到 0.5301——那是**好消息**，证明 LLM 因子有用。
如果 P7 跑出来 AUC 从 0.5131 下降到 0.4901——那是**坏消息**，但也是**有用的信息**：LLM 因子有害，下次别加。
我们 P7 跑出来 AUC 没动 + importance=0 + 根因是数据太稀——**这也是有用的信息**：瓶颈不在算法在数据。

**三种结果都比"装作 LLM 有用、继续投入"要有价值**。

这正是你 6 个月前设 `auditor` 那一刻在守护的东西——**不管结果好坏，都说真话**。从报告 03 的 AUC 0.51 "不装水准高"，到报告 04 的 14 笔 +5.92% "不包装成 alpha"，到现在报告 07 的 importance=0 "不掩盖"——**这是项目最值得自豪的事**。

走到这里，Stage 2 的对照实验跑完了。接下来 P8 是 Stage 2 收尾（跟 Stage 1 的 P5 类似——code-reviewer 深审 + verifier 端到端再跑一遍）。**Stage 2 即将以一个诚实的 negative finding 收官**——这不是失败，是一个真正的研究成果。
