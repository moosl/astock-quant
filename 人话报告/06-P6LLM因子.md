## 三、LLM 因子干什么——用「7×24 不睡的分析师」类比

### 3.1 什么是 LLM、什么是 LLM 因子（再讲一次）

> **名词解释·LLM（大语言模型）**：像 ChatGPT、Claude 这种 AI——能看懂文字、能写文字。
>
> **名词解释·LLM 因子**：让 AI 去读每天的新闻、研报、公告这些**文字内容**，把「这条新闻对这只股票是利好还是利空」的判断，**转成一个数字**，作为一种"因子"喂给量化模型。
>
> 关键点（报告 01/05 说过、再强调一次）：**AI 不直接预测股价**。AI 只做"文字 → 数字"的翻译工作，真正的预测还是 LightGBM 干。

打个最直观的比方：

> 想象你雇了一个**7×24 小时不睡觉、不抱怨、不要工资的分析师**（其实就是 Claude）。它的任务很简单——**每天把当天关于这只股票的所有新闻看一遍，然后给个分数**：
>
> - 「茅台业绩超预期增长」→ **+1 分**（强利好）
> - 「茅台监管处罚暴雷」 → **-1 分**（强利空）
> - 「茅台董事会换届，主席继续连任」→ **0 分**（中性消息）
> - 「市场传闻茅台或调整经销商政策」→ **+0.5 分**（弱利好）/ **-0.5 分**（弱利空）（取决于内容）
>
> 每条新闻 AI 还会给一个**自信度**——「我对这条判断有多自信？」。从 0（完全瞎猜）到 1（很有把握）。

### 3.2 五级情绪框架是什么

> **名词解释·五级情绪**：项目里把 AI 的判断**强制压缩到 5 个档位**：-1 / -0.5 / 0 / +0.5 / +1。这叫"离散化"——不让 AI 输出 0.347 这种过分精细的小数，因为它的判断本来就达不到那种精度。**5 档已经够区分了**。

为什么不让 AI 自由输出 0~1 任意数字？因为：
- AI 给出 0.347 这种数字时，它**没法解释为什么不是 0.348**。强制 5 档反而更稳定。
- 5 档已经能区分「强利好 / 弱利好 / 中性 / 弱利空 / 强利空」——研究新闻情绪一般也就这个分辨率。

### 3.3 一天有好几条新闻怎么办——加权平均

A股 一只票一天通常有 0 ~ 5 条相关新闻不等。**那 AI 同一天给的 N 个分数怎么合成一个数**？

用 **confidence 加权平均**：

```
当天的最终情绪分 = Σ(每条新闻的分数 × 这条的自信度) ÷ Σ(自信度)
```

> **名词解释·加权平均**：不简单求平均，而是按"重要程度"给不同的数据加权。AI 自信度高的判断，在合成时占更大比重；自信度低的判断，影响比较小。
>
> 跟"听 3 个专家意见、其中一个明显说『我不太确定』"是一样的道理——你不会把他和肯定的那两位平等对待。

举个例子，茅台某天有 2 条新闻：

| 新闻 | AI 给的情绪分 | AI 给的自信度 |
|---|---:|---:|
| 「茅台业绩超预期」 | +1.0 | 0.9（很有把握）|
| 「市场担忧白酒消费走弱」 | -1.0 | 0.1（不太确定）|

如果**简单平均** = (1.0 + (-1.0)) / 2 = **0** ——错！

**confidence 加权** = (1.0 × 0.9 + (-1.0) × 0.1) ÷ (0.9 + 0.1) = (0.9 - 0.1) ÷ 1.0 = **+0.8** ——对！

这正确反映了「主要信号是强利好，次要的弱利空稀释了一点点」。

**没新闻的日子怎么办**？返回 NaN（空值，不强行填 0）。这是延续报告 03 讲过的"P3a 因子层故意保留 NaN"纪律——**别用假数据骗模型**，让 LightGBM 自己处理空值。

## 四、三个关键设计

P6 这次有 3 个设计上很讲究的决定，每一个都讲清楚"为什么这么做"。

### 4.1 默认关闭——给你的钱包加把锁

**这次最重要的设计之一**。LLM 因子默认是关的，**你必须显式打开开关才会真的去调 Claude**。

> **名词解释·env var（环境变量 / environment variable）**：电脑里"一些设置"的存放地。代码不写死、由环境变量决定——好处是**同一份代码在不同人电脑上行为可以不同**。比如你电脑上设了"开 LLM"、CI 服务器没设，那 CI 永远不会调 LLM。

具体怎么开：

```bash
export ANTHROPIC_API_KEY=sk-ant-xxxxx     # 你的 Claude API 钥匙（必需）
export ENABLE_LLM_FACTOR=1                 # 开关（设为 1 才打开）
```

不设这两个的话：
- 跑 `run_direction()` 还是用老的 25 个量价/财务/资金流因子
- **一次 LLM 调用都不会发生**
- 你账户上一毛钱不会被烧

#### 为什么必须默认关

3 个理由，每一条都重要：

1. **保护你的钱包**：万一你或我手滑触发了循环、跑了 1 万次 LLM 调用——你的 API 账户被刷光。**默认关 = 必须主动打开**，这是一道"保险丝"。
2. **不破坏 Stage 1**：报告 05 强调过 60 个测试全过、metrics 一个数字都没漂移。如果默认开了 LLM，CI（自动测试系统）每次跑都会去调 LLM——又花钱又慢，关键是**没 API 钥匙的人根本跑不起来**。默认关 = 别人 clone 这个项目，啥都不配置直接 `pytest` 就能跑通。
3. **学习路径上更安全**：你想"先看看默认行为再决定要不要试 LLM"——默认关让你能这么做。

#### 验证：默认关时 Stage 1 一字不变

auditor 实测：**设 `ENABLE_LLM_FACTOR=0`（或不设）时**，跑整条流水线出来的所有数字，跟 Stage 1 收官版一字不差：

| 指标 | P6 实测（关 LLM）| Stage 1 收官 | 一致？ |
|---|---:|---:|:---:|
| 因子数量 | 25 | 25 | ✅ |
| AUC | 0.5131337782587783 | 0.5131337782587783 | ✅ **完全一致** |
| 训练集行数 | 25283 | 25283 | ✅ |
| 验证集行数 | 5780 | 5780 | ✅ |
| 回测天数 | 193 | 193 | ✅ |

这意味着——**P6 是个纯加法**。你不打开开关，老成果一字不变；你打开开关，多一列新因子；想关又能秒切回去。

### 4.2 多供应商可切换——不绑死 Claude

这次默认用的是 Claude（Anthropic 公司的 LLM）。但**factor-engineer 写代码时刻意没把 Claude 写死**——以后想换 OpenAI、DeepSeek、Kimi 等任何一家，**改一个环境变量就行**：

```bash
export LLM_PROVIDER=openai      # 或 deepseek、kimi 等
export LLM_MODEL=gpt-5-mini     # 对应的模型名
```

#### 为什么不绑死一家

3 个原因：

1. **价格会变**：今天 Claude 便宜，明天 OpenAI 出新模型更便宜，再后天 DeepSeek 开源版几乎免费。**绑死 = 后悔**。
2. **效果可对比**：你想知道「Claude vs GPT 谁读新闻读得准」——切换一行配置就能跑两版对照。这是研究的乐趣。
3. **延续报告 01 的"分层"思想**：报告 01 讲过"模块化 = 哪块坏了换哪块"。LLM 也是个"模块"，今天叫 Claude，明天可以换。

#### 怎么做到的——「Protocol（协议）」是什么

> **名词解释·Protocol（协议）**：Python 里一个特别灵活的"接口约定"工具。
>
> 打个比方：你开个外卖店，规定"任何外卖员必须会两件事——`接单()` 和 `送货()`"。**你不关心他穿什么衣服、骑什么车、是男是女**。只要会这两件事，他就是你的外卖员。
>
> Protocol 就是这种"我只管接口、不管具体长啥样"的约定。在 P6 里，规定"任何 LLM 客户端必须实现两个方法：`chat`（聊天）和 `chat_json`（结构化输出）"。Claude 客户端实现这两个就能用；OpenAI 客户端实现这两个也能用；甚至**测试时用的 mock（假的）客户端**实现这两个，照样能用。

#### 顺便做了件特别讲究的事——不引重型框架

报告 01 讲过一个"方案 A 瘦身版"——**研读开源框架但不直接 import 它们**。这次 factor-engineer 又一次贯彻了这个原则。

他研读了 `TradingAgents-astock` 这个开源项目的 LLM 封装——发现它用了一个叫 **`langchain`** 的重型框架。langchain 啥都能干，但**对我们这种「就让 AI 给一条新闻打分」的简单场景，太重了**——背一堆用不上的工具、还有依赖兼容性问题。

所以 factor-engineer **完全没 import langchain**，自己用 Protocol 写了一个最小客户端（核心代码 50 行左右）。auditor 实测 grep 验证：**整个 P6 代码里 0 处真实 import langchain**，只在注释里说"研读出处是 langchain，但我们没用"。

> **名词解释·依赖（dependency）**（报告 01 讲过）：你项目"装"进来、运行时离不开的别人的代码包。**依赖越少，项目越轻、越稳定、越容易理解**——你看到的每一行代码都是项目自己的，没有黑盒。

这跟报告 05 强调的"工程纪律"是一脉相承的。

### 4.3 缓存机制——一条新闻一辈子只让 AI 读一次

报告 02 讲过 P2 数据缓存（"自己家冰箱"的比喻）。P6 给 LLM 调用配了**一样的缓存机制**，但意义更大——因为 LLM 调用是**真烧钱**的。

#### 缓存怎么工作

```
你跑流水线 → 看到一条新闻「茅台业绩超预期」
            ↓
            先查"AI 答题本"（缓存）：之前 AI 看过这条吗？
            ↓
   ┌────────┴─────────┐
   ↓                  ↓
看过 → 直接读答案    没看过 → 调一次 LLM → 把答案存进答题本
   ↓                  ↓
   返回 +1.0           返回 +1.0
   花费 0 元           花费 ~0.001 美元
```

#### 这个缓存的关键：**怎么判断"是不是同一条新闻"**

每条新闻被分配一个唯一 ID（叫 `news_id`）：

- **有 URL 的新闻**：用 URL 算一个唯一码（URL 是稳定的，同一条新闻在哪都是这个 URL）
- **没 URL 的新闻**：用「股票代码 + 日期 + 标题」算一个唯一码（兜底）

**所以**：你第一次跑 → AI 读 9000 条新闻 → 假设花 1.5 美元；**第二次跑 → 答题本全命中 → 0 调用、0 花费**。你之后无论调阈值、调模型超参、跑多少次回测、做多少对照实验——**只要新闻没变，LLM 调用就是 0**。

#### 跟报告 02 的缓存联动

报告 02 那个缓存（数据缓存）让"拉数据"从 32 秒降到 0.07 秒（460 倍提速）。这次 LLM 缓存的提速比那个更夸张——**第二次跑的成本从 1.5 美元降到 0 美元，省了 100%**。

## 五、API key 安全——auditor 专门点赞的工程文化

> **本节是 auditor 在审核报告里专门强调"亮点"的部分。讲讲为什么**。

### 5.1 什么是 API key、为什么要严防

> **名词解释·API key（API 钥匙）**：你向 Claude / OpenAI 之类的服务申请的"身份证 + 信用卡"，长得像 `sk-ant-xxxx...` 这样的一串字符。
>
> **它的两个属性**：
> 1. **它代表你**——拿着它的人就是你
> 2. **它能花你的钱**——每次调用 LLM 都按你的账户收钱
>
> **所以它泄漏出去 = 钱包被偷**。常见泄漏方式：
> - 提交到 git 仓库被人 clone 看到
> - 写进日志被人 SSH 看到
> - 截图发到群里被人 OCR 提取
> - **写进缓存文件**被人 dump 看到

### 5.2 P6 给 API key 加的三道防线

像不像报告 04/05 讲的"三道防 look-ahead 防线"？这次也是三道，但是防钱包被偷的。

```
第一道：env var 唯一入口
       ─→ API key 只从环境变量读，不写在代码里、不存配置文件里
              ↓
第二道：代码里写明纪律
       ─→ 每个相关文件顶部写「绝不硬编码、绝不写进缓存、绝不打印到日志」
              ↓
第三道：测试断言守住
       ─→ 自动化测试里直接搜：缓存文件里有没有 "ANTHROPIC_API_KEY" 字串？
                              有没有 "sk-" 字串？ 有 → 测试当场报警
```

#### 第三道防线是 auditor 重点点赞的——把"安全审计"写进了测试

听起来抽象，看一段实际的测试代码就明白了：

```python
def test_cache_file_layout():
    # ... 跑一次 LLM 因子计算，让它产生缓存文件 ...

    serialized = cache_file.read_text()       # 把缓存文件内容读出来
    assert "ANTHROPIC_API_KEY" not in serialized   # 缓存里绝不能有 API key
    assert "sk-" not in serialized                  # 缓存里绝不能有 key 前缀
```

意思是：**这个测试每次跑都会扒开缓存文件检查里面有没有泄漏 API key 的迹象**。一旦未来有人不小心改坏代码、把 key 写进了缓存——这个测试当场红，CI 立刻报警。

#### 这件事的意义（auditor 点赞的原因）

工程上，**安全审计通常是"靠人记得"**——code review 的时候人盯着代码看一遍，靠人脑判断"这里有没有泄漏 key"。**人脑会忘**。

**写进测试 = 永远不会忘**。这是把工程纪律**自动化、机制化**——跟报告 03 讲的"pytest 长期值守"是同一个思想，只是这次值守的不是数学正确性，**是钱包安全**。

### 5.3 缓存里到底存什么——开了看一眼

每个缓存文件只存 5 个字段，**严格清单**：

| 字段 | 是什么 | 大小 |
|---|---|---|
| `sentiment` | AI 给的情绪分（-1/-0.5/0/+0.5/+1）| 1 个数字 |
| `confidence` | AI 给的自信度（0~1）| 1 个数字 |
| `reason` | AI 简短解释「为什么打这分」| 截到 200 字 |
| `title` | 新闻标题（备查）| 截到 120 字 |
| `source` | 新闻来源（哪个网站）| 短字符串 |

**严格不存的**：
- API key（已审）
- 你发给 AI 的完整 prompt（防止 prompt 里万一写了敏感信息）
- AI 返回的原始 response（防止 response 里 echo 了 prompt）
- 调用参数（温度、最大 token 等——这些不敏感但也没必要存）

## 六、Stage 1 字节级不变——这个项目的"加法纪律"

这一节强调 P6 最重要的兼容性承诺。

### 6.1 什么叫"字节级不变"

> **名词解释·字节级不变 / bit-exact**（报告 03 讲过，再提一次）：两边算出来的数字**一个 bit 都不差**，完全相同。

P6 给项目加了一整套新东西（4 个新文件、1 个新因子、23 个新测试），但 auditor 实测确认：**默认关闭时，跑流水线出的所有数字，跟 Stage 1 收官版完全一字不差**——AUC、训练行数、验证行数、回测笔数、收益、夏普、回撤……14 个核心指标，**没一个漂移**。

### 6.2 为什么这事极其重要

想象你做一个升级。**最糟糕的情况是**——"我加了个新功能，老功能附带变了"。这种事在软件工程里超常见：
- 你新增了一个因子，结果发现老 25 个因子里有一个被你不小心改名了
- 你重构了一下代码结构，结果训练集行数从 25283 变成 25281（少了 2 行）
- 你引入了新库，结果某个老库的默认行为被覆盖了

**这种"附带变化"会让你彻底搞不清「现在的版本到底跟之前比变了什么」**——下次你跑出 AUC 0.52，你不知道是 LLM 因子真有用还是只是某个隐藏改动碰巧的副作用。

P6 这次的"加法纪律"——**默认关时所有数字 bit-exact 一致**——保证了：
- 你打开开关时看到的任何 AUC 变化，**100% 是 LLM 因子带来的**，不是其他东西
- 你随时能"切回"老版本（关掉开关），结果跟之前完全一致
- Stage 1 的所有成果（5 份人话报告、12 份审核、60 个测试）**全部继续有效**

### 6.3 这跟前面讲过的"分层 + 插槽"是一回事

## 八、接下来 P7：LLM 因子的"考试"

P6 把 LLM 因子**接进了系统**，但**它到底有没有用、能不能提升模型预测能力**——这是 P6 *没* 回答的问题。

下一阶段 **P7 对比验证**才回答这件事，由 verifier 接力。

```
P6 ──→ [P7 对比验证] ──→ P8 审查 + 验证 ──→ Stage 2 收尾
        ▲▲▲▲ 下一站
```

### 8.1 P7 怎么验证

很简单——**做对照实验**：

```
A 组（baseline）：关 LLM 因子，跑流水线 → 看 AUC / 回测收益等
                         ↓                ↓
B 组（实验组）：开 LLM 因子，跑流水线 → 看 AUC / 回测收益等
                         ↓                ↓
                            对比两组数字
                            ↓
                  B 组明显好 → LLM 因子真有用
                  B 组差不多 → LLM 因子没贡献
                  B 组反而差 → LLM 因子在帮倒忙（也是个发现！）
```

这跟报告 04 讲的"三组对照"思路一脉相承——**绝对收益数字没意义，得跟对照组比起来好多少才算数**。

### 8.2 P7 之前 auditor 给的两个提醒

auditor 在 P6 审核里给 verifier 留了几条 actionable advice：

#### 提醒 1：先小池子试一下

> **名词解释·小池子（small universe）**：先用 2-3 只股票试，而不是直接上 30 只。
>
> 类比：你要给一群人做新菜——肯定先做 1 份试试味道再决定要不要给全场上菜，**不能直接做 100 份发现难吃**。

**先用 3 只票（比如 600519 / 000858 / 000001）跑端到端**，确认两件事：
- LLM 调用真能跑通（API key 配对了、网络通了、新闻能拉到、AI 真能给出 JSON 格式回答）
- token 消耗在预期之内（不会跑一半发现 token 烧了 $10 还没跑完）

然后再扩到 30 只票跑完整对照。

#### 提醒 2：注意 NaN 比例

> **名词解释·NaN 比例（nan_ratio）**：因子值里"空值"占多少比例。
>
> 之前报告 03 讲过——25 个因子里有 7 个 NaN 比例超 80%（PE/PB + 资金流），模型基本学不到信号。LLM 因子也有这个问题——**有些日期或股票本来就没新闻**。

A股 新闻覆盖**早期可能很稀疏**。30 只票 × 4.4 年 ≈ 1.3 万个交易日点位，但可能有大量日子根本没新闻。**LLM 因子在那些日子就是 NaN**——LightGBM 能处理 NaN，但**有效样本量**会缩水。

P7 要先量一下：「LLM 因子在 31603 行里，有多少行是 NaN？是 30% 还是 90%？」如果 NaN 太多，需要先扩展新闻源（或者拉长 lookback 窗口，让今天的"情绪分"用过去几天的新闻加权出来）。

### 8.3 P7 后还有 P8

P8 是 Stage 2 的收尾，跟报告 05 讲的 Stage 1 P5 类似——code-reviewer 深审 LLM 相关代码 + verifier 端到端再跑一遍。然后 Stage 2 就收官。

> **名词解释·token（令牌）**：LLM 处理文字的最小单位。一段中文大概 1 字 = 1.5 token。LLM API 按 token 计费——你发给它 500 个 token、它回 50 个 token，按这个数字收钱。

## 九、一页纸总结

- **P6 干了啥**：Stage 2 第一个实质阶段——把"让 AI 读新闻"这条新支路真的接进了系统。在因子层加了一种新因子叫 `LLMNewsSentiment`，跟之前的 25 个量价/财务/资金流因子**完全平级**，一起喂给 LightGBM。
- **LLM 因子是什么**：让 Claude 读每条新闻，给一个 -1 ~ +1 的情绪分（五级离散：-1/-0.5/0/+0.5/+1）+ 一个自信度（0~1）。同一天多条新闻 → confidence 加权平均合成一个数。没新闻 → NaN，不强行填 0。
- **三个关键设计**：
  - **默认关闭**：要主动设 `ENABLE_LLM_FACTOR=1` 才打开。保护你的钱包 + 不破坏 Stage 1 + CI 友好。
  - **多供应商可切换**：默认 Claude，将来换 OpenAI / DeepSeek / Kimi 改一个环境变量。用 Python Protocol（极轻量接口）实现，不引重型框架 langchain。
  - **缓存严格去重**：每条新闻有唯一 ID（URL 优先 / 标题 hash 后备），落盘 JSON。**第二次跑 LLM 调用数 = 0、成本 = $0**。
- **API key 安全 = 3 道防线**（auditor 专门点赞）：
  - 第一道：env var 唯一入口，不写代码里
  - 第二道：代码里写明纪律（绝不硬编码 / 绝不写缓存 / 绝不打印日志）
  - 第三道：**测试自动 grep 缓存文件**，搜到 `ANTHROPIC_API_KEY` 或 `sk-` 字串当场报警。这种把"安全审计自动化"是工程上罕见的好实践。
- **Stage 1 字节级不变**：默认关闭时，AUC=0.5131337782587783 / 训练 25283 / 验证 5780 / 回测 193 天 / 0 交易 等 14 个指标跟 Stage 1 收官完全一字不差。这意味着 P6 是个**纯加法**——加了不破坏老的，关了等于没加，开了又能随时关回去。
- **auditor 第 7 次实战 PASS 稳定**：85 测试全过（62 旧 + 23 新）、ruff clean、scope 严守只动 7 个文件、6 个工程亮点（Protocol 设计 / 安全测试 / JSON 解析鲁棒 / env 三档全测 / 延迟 import / TODO 透明）。
- **接下来 P7 对比验证**：A 组关 LLM vs B 组开 LLM，跑同一条流水线对比 AUC / 回测。这才是"LLM 因子到底有没有真本事"的考试。auditor 建议**先 3 只票小池子 smoke test、再扩到 30 票**；注意先量一下 LLM 因子在历史数据上的 NaN 比例，避免有效样本量大幅缩水。
