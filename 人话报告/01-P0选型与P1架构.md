## 一、这个项目到底在做什么

**一句话：我们在造一个「用机器学习预测 A 股」的练习系统。**

把它想成你要造一台「炒股预测机器」。你往里喂股票的历史数据，它学习规律，然后告诉你它的判断。但注意定位——**这是台「学习机」，不是「印钞机」**：目标是能跑起来、代码能看懂、能自己改着玩，用来搞懂量化到底是怎么回事，不是拿去实盘梭哈。

这台机器具体要会四样本事（我们叫「4 类预测目标」）：

| 它要回答的问题 | 行话叫法 | 大白话 |
|---|---|---|
| 这只股票明天会涨还是跌？ | 涨跌方向（二分类） | 像抛硬币猜正反，只要答「涨」或「跌」 |
| 这只股票未来能涨/跌多少个点？ | 收益率回归 | 不光猜方向，还要猜「幅度」 |
| 这一篮子股票里，哪几只最值得买？ | 选股排序（Top N） | 给一堆股票排个名，挑出最靠前的 |
| 现在这只股票该买、该卖、还是该拿着？ | 买卖信号 | 直接给操作建议 |

> **名词解释·机器学习**：让计算机自己从大量历史数据里「找规律」的技术。你不用手把手教它「股价跌了 5 天就会反弹」这种规则，你只要把几年的数据丢给它，它自己总结。这跟「人写一堆 if-else 规则」是两条路。

整个系统的数据是这样流动的，一条流水线：

```
拉数据  →  算因子  →  打标签  →  训练模型  →  回测  →  出信号
（原料）  （特征）   （答案）   （学习）    （考试）  （结论）
```

后面会逐个解释每个环节。先记住：**这是一条流水线，原料从左边进，结论从右边出。**

## 三、P0 发生了什么：选「站在谁的肩膀上」

### 3.1 P0 在干的事

你之前自己去 GitHub 上挑了 3 个开源项目，发给了团队。P0 这个阶段，团队（document-specialist，文档调研）做的事就是：**把这 3 个项目的代码一行行读完，告诉你它们到底能不能用、怎么用。**

> **名词解释·框架 / 开源项目**：别人已经写好、免费公开的一套代码。用它的好处是「不用从零造轮子」。这 3 个就是别人造好的「股票分析系统」。

那 3 个项目是：
- **TradingAgents-astock** —— 专门为 A 股改造过的「AI 投研团队」
- **ai-hedge-fund** —— 一个很有名的「AI 对冲基金」项目
- **TradingAgents** —— 上面第一个的「原版」（针对美股的）

### 3.2 P0 的意外发现（重点，影响整个项目走向）

团队读完代码，发现了一件**计划外的事**：

> **这 3 个项目，没有一个是「用机器学习预测股票」的。它们全都是「让一群 AI 角色互相辩论、吵出一个买卖决定」的系统。**

打个比方。我们本来以为这 3 个里至少有一个是「预测机器」——你喂数据，它算出涨跌。结果打开一看，3 个全是**「AI 圆桌会议」**：里面有 7 个、甚至 13 个 AI 分别扮演「技术分析师」「基本面分析师」「巴菲特」「芒格」……每个 AI 写一份研报，然后多空双方辩论，最后一个「基金经理 AI」拍板买还是卖。

这两种东西**完全不一样**：
- **AI 圆桌会议**：靠「AI 读资料 + 讲道理」出决定。它不「学习」,每次都是现场分析。
- **机器学习预测**：靠「从几年历史数据里自己总结的规律」出判断。它要「先学习、再预测」。

而我们这个项目的核心——计划里白纸黑字写的「用 LightGBM 这类机器学习模型预测」——**那 3 个框架里一个都没有**。它们的代码里连 `LightGBM`、`LSTM` 这些机器学习工具的影子都找不到。

> **名词解释·LightGBM**：一个非常流行的机器学习工具（属于「树模型」家族）。你可以把它想成一个特别会「在海量数据里找规律」的引擎，吃进去几十种股票特征，吐出来一个预测。量化界用它的人极多，因为它又快又准、还不太挑数据。后面我们的「预测大脑」就是用它造。

### 3.3 这个意外的影响：核心得自己造，框架只能「拆零件」

既然 3 个框架都没有我们要的「预测大脑」，那结论就很清楚了：

**「机器学习预测」这个最核心的部分，团队得自己写。这 3 个框架不能整个搬来用，只能当「零件供应商」——从里面拆有用的零件。**

好在拆出来一看，零件还挺值钱。团队盘了一下，这 3 个框架里能拆的好零件有：

| 能拆的零件 | 来自哪个框架 | 干什么用 |
|---|---|---|
| **A 股数据接入模块** | TradingAgents-astock | 一段已经写好的代码，能直接把 A 股的行情、财务数据拉下来、还自带「缓存」和「防作弊」机制。这是最值钱的零件。 |
| **回测引擎** | ai-hedge-fund | 一套完整能跑的「模拟炒股考试系统」。3 个框架里唯一一个真能跑的。 |
| **项目结构设计** | ai-hedge-fund 的 v2 版 | 不是代码，是「一个干净的量化项目该怎么分文件夹」的范本。我们的图纸（P1）就参考了它。 |
| **因子计算的零碎代码** | 两个框架都有 | 一些算技术指标（RSI、MACD 之类）的现成小代码段 |
| **AI 读新闻的封装** | TradingAgents-astock | 留到第二块（P6）「让 AI 读新闻」时用 |

> **名词解释·回测**：拿历史数据给模型做「模拟考试」。比如假装现在是 2023 年初,让模型按它的预测去「模拟买卖」,一直跑到 2024 年底,看它最后是赚是亏、赚多少、中间最大亏过多少。**回测是检验「这个模型到底行不行」的唯一办法**——总不能拿真钱去试错。

> **名词解释·因子**：影响股价的某一个「特征」或「信号」。比如「这只股票最近 20 天涨了多少」是一个因子，「它的市盈率高不高」是另一个因子,「最近主力资金是流入还是流出」又是一个因子。**机器学习模型吃的「原料」就是一大堆因子**。因子造得好不好，直接决定模型聪不聪明。

**所以 P0 的最终结论一句话**：3 个框架定位看错了（不是预测系统，是 AI 辩论系统），但它们身上有值钱的零件。我们的策略从「选一个框架来用」变成「从 3 个框架里拆零件，自己拼一台预测机器」。

## 四、P1 发生了什么：画房子的图纸

P0 解决了「站在谁肩膀上」，P1（architect，架构设计）解决「**这房子怎么盖**」——在写代码之前,先把整个项目的结构、文件怎么分、模块怎么配合，全部设计清楚。

P1 做了 3 个关键决定，逐个讲。

### 4.1 决定一：选「方案 A 瘦身版」——把框架当「参考答案」，不当「依赖」

P0 报告留了个选择题给 P1：到底怎么用那 3 个框架的代码？两个极端选项：

- **方案 A（原版）**：把框架整个「装」进我们项目，直接调用它们的代码。**省事**，但等于把一大堆你看不懂的别人的代码绑在身上。
- **方案 B（纯自建）**：完全不碰框架代码，每一行自己从零写。**最干净**，但工作量极大，还容易重复踩别人踩过的坑。

> **名词解释·依赖**：你的项目「装」进来、运行时离不开的别人的代码包。依赖越多，项目越重，出了 bug 越难查——因为 bug 可能藏在别人的代码里，你根本看不见。学习型项目尤其怕这个。

架构师选了**中间路线，叫「方案 A 瘦身版」**。核心思想用一个比喻：

> **把那 3 个框架的成熟代码当成「教辅书上的标准答案」——你认真研读、理解透,然后用自己的话把解题过程重新誊写一遍到自己的作业本上。但你不会把整本教辅书钉在自己的作业本里。**

具体说就是：
- 框架代码**只看不装**。要写「回测引擎」时，把 ai-hedge-fund 那套打开,研究它怎么逐日推进、怎么算收益,理解了，然后**用自己的话在我们项目里重写一遍**。
- 但**不把那些框架作为「依赖」装进来**。我们项目第一块只装真正必要的、轻量的工具（pandas、LightGBM 这些行业标配）。像 LangGraph 那种重型的「AI 辩论框架」一概不装。

**为什么这么选**（这是关键，你要的就是「为什么」）：
- **不选纯方案 A**：因为那 3 个框架的核心是「AI 辩论」，跟我们要的「机器学习预测」是两条路,装进来纯属背包袱、堆一堆用不上的概念。而且里面有的「依赖」本身就是摆设(比如 TradingAgents-astock 声明了一个回测库,结果整个项目根本没用过它)。
- **不选纯方案 B**：因为 ai-hedge-fund 那套回测引擎是经过验证、真能跑的,自己从零硬写很容易在「防作弊」这种细节上栽跟头。有标准答案可以参考,何必闭着眼睛瞎写。
- **选瘦身版**：既享受「有标准答案参考、少踩坑」的好处,又保住「项目里每一行代码都是自己的、看得懂、改得动、能调试」——这正好是「学习型项目」最该要的。

> **名词解释·防作弊（look-ahead bias，未来函数）**：量化里的头号大坑。简单说就是**模型不小心偷看了「未来的答案」**。比如你让模型预测「明天涨不涨」,结果训练时不小心把「明天的股价」也喂给它了——那它当然预测得神准,但这是作弊,拿到真实环境里立刻原形毕露。整个项目要时时刻刻防着这个,P1 专门设了两道防线。

### 4.2 决定二：把项目拆成 9 个「分层模块」+ 一份「数据契约」

架构师把整个项目拆成了 9 个文件夹（叫「模块」），每个模块只管一件事。还是用流水线的比喻，从原料到成品：

| 模块 | 大白话职责 | 流水线位置 |
|---|---|---|
| **config**（配置） | 「设置面板」——选哪些股票、回测从哪天到哪天、本金多少，集中放一处 | 流水线的「参数旋钮」 |
| **data**（数据层） | 把 A 股的行情、财务、资金流数据真正拉下来 | ① 进原料 |
| **factors**（因子层） | 把原始数据加工成一个个「因子」(特征) | ② 加工特征 |
| **labels**（标签层） | 给历史数据「打标准答案」——比如这天之后 5 天确实涨了，标个「1」 | ③ 准备答案 |
| **models**（模型层） | 「预测大脑」——拿「特征 + 答案」训练 LightGBM,学完会预测 | ④ 学习 + 预测 |
| **backtest**（回测层） | 「模拟考试系统」——让模型在历史里模拟炒股，算出赚亏 | ⑤ 考试 |
| **signals**（信号层） | 把模型冷冰冰的预测数字，翻译成人能看懂的「买/卖/持有」 | ⑥ 出结论 |
| **pipeline**（编排层） | 「总开关」——把上面 6 步串成一条，一个命令从头跑到尾 | 把①~⑥串起来 |
| **contracts**（数据契约） | 不是流水线的一站，是「**统一的零件接口标准**」 | 贯穿全程 |

**为什么要「分层」**（又是「为什么」）：

想象你修一台机器。如果它是一整块焊死的铁疙瘩，坏了哪里你根本没法修。但如果它是**乐高式的、一块一块拼起来的**——数据这块坏了，你只动数据这块,别的不用碰;想换个更好的预测模型,只换「模型」那块,流水线其他部分原封不动。

**分层 = 把大问题切成一个个小问题,每个模块只管自己那一摊。** 对学习型项目特别重要:你想搞懂「因子是怎么算的」,只要去读 factors 那个文件夹,不会被其他东西干扰。架构师在 P1 文档里有句话点透了——「学习型项目成败在结构」。

**那「数据契约」又是什么**：

> **名词解释·数据契约**：模块和模块之间「交接东西」时,事先约定好的**统一格式**。

打个比方:流水线上「数据层」要把一根 K 线数据交给「因子层」。如果没有统一格式,数据层可能传个张三格式,明天心情好又传个李四格式,因子层就崩溃了。**数据契约就是规定:「所有人交接 K 线,都必须用这一个标准格式,字段叫什么、什么类型,全统一」。** 这样任意两个模块之间都能严丝合缝地对接。技术上这个用一个叫 Pydantic 的工具来定。

> **名词解释·Pydantic**：Python 里一个专门用来「定义并强制检查数据格式」的工具。你用它规定「一根 K 线必须有日期、开盘价、收盘价……」,之后但凡有人传个缺胳膊少腿的数据进来,它当场报错,不会让脏数据偷偷流到下游。相当于流水线上每个交接口都装了个「质检员」。

### 4.3 决定三:两个「聪明设计」

P1 里还有两个设计,不是「必须这么做」,而是「这么做很巧」,值得单独讲。

**聪明设计 ①:4 类预测目标,共用一套地基**

还记得开头说的吗?这系统要会 4 样本事(猜涨跌、猜幅度、排序选股、出买卖信号)。

笨办法是:为这 4 样本事各造一条完整流水线,4 套数据层、4 套因子层…… 那是巨大的重复劳动。

架构师的巧办法:**4 样本事「共用」前半条流水线,只在后半段分叉。**

```
        【4 类目标 100% 共用,一模一样】      【这之后才按目标分叉】
config → data → factors  ────────────────→  labels → models → signals
（选股）（拉数）（算因子）                      （打答案）（预测）（出信号）
```

道理很简单:不管你是要「猜涨跌」还是「排序选股」,**喂给模型的原料(因子)都是同一套**。所以「拉数据、算因子」这前半截,造一次,4 个目标一起用。只有到了「打什么样的答案、用什么模型、出什么形式的结论」这后半段,才按目标类型(技术上叫 `target_type`)分开处理。

**好在哪**:省掉 75% 的重复劳动;而且第一块我们只先把「猜涨跌」这一条做透,以后要补另外 3 个,**地基不用动,只在少数几个文件里加代码就行**——不用推倒重来。

**聪明设计 ②:给「让 AI 读新闻」提前留好接口**

我们项目分两大块,第二块(P6)才会加「让 AI 读新闻、把舆情变成因子」这个升级包。

但架构师在第一块画图纸时,**就提前给这个升级包留好了「插槽」**。

> **名词解释·LLM 因子**:LLM 就是大语言模型(像 ChatGPT、Claude 这种 AI)。「LLM 因子」的意思是——让 AI 去读新闻、研报、公告这些**文字**,然后把「这条新闻是利好还是利空」「情绪有多强」这种判断,变成一个数字。这个数字就成了一种「因子」,跟「最近涨了多少」那些数字因子一样,一起喂给预测模型。
>
> 注意:在我们这套设计里,**AI 不直接预测股价**,AI 只负责「把文字变成一个因子数字」,真正的预测还是交给 LightGBM。

留接口的具体做法:架构师让「AI 读新闻产出的因子」和「普通数字因子」长成**一模一样的形状**(都属于同一个「因子」家族、产出同一种数据契约)。这样一来,等第二块真要接 AI 读新闻时,对流水线的其他部分来说——**它只是「又多了几个因子」而已**,数据层、模型、回测、信号,一行代码都不用改。

**好在哪**:这叫「平滑接入」。很多项目是后期硬塞新功能,塞得整个系统千疮百孔。我们提前留好插槽,第二块的升级包「啪」一下就插上了,不动旧的东西。这是「先想清楚再动手」的典型好处。

### 4.4 P1 还做了一件实事:骨架已经搭出来了

P1 不只是写文档,architect 已经按图纸把**项目骨架**真正建出来了——9 个模块文件夹、几十个 `.py` 文件、那份数据契约,全都创建好了,而且通过了「能正常加载」的检查。

## 五、接下来要做什么:P2 数据管道

P1 过关后,下一阶段是 **P2 数据管道**,由 data-engineer(数据工程)负责。

## 六、一页纸总结

- **项目是什么**:造一台「用机器学习预测 A 股」的**学习用**系统,要会猜涨跌、猜幅度、排序选股、出买卖信号 4 样本事。
- **现在在哪**:P0、P1 已完成(都是动脑不动手的阶段);下一步 P2 开始真正写代码。
- **P0 的意外**:你挑的 3 个开源框架,定位看错了——它们是「AI 辩论决策」系统,不是「机器学习预测」系统,核心的预测大脑一个都没有。结论:核心自己造,框架只拆零件用(数据接入、回测引擎、项目结构是拆出来最值钱的零件)。
- **P1 的 3 个决定**:
  1. 选「方案 A 瘦身版」——框架代码当「标准答案」研读后自己重写,但不把框架当「依赖」装进来。既少踩坑,又保证每行代码自己看得懂。
  2. 项目拆成 9 个「乐高式」分层模块 + 一份「统一接口标准」(数据契约)。分层 = 大问题切小,坏了好修、想学好读。
  3. 两个聪明设计:4 类预测目标「共用前半条流水线」省掉重复劳动;给第二块「AI 读新闻」提前留好插槽,将来平滑接入不动旧代码。
- **P1 也搭好了「毛坯房」**:骨架文件全建好了,但还是空的,P2 开始装修。
- **下一步 P2**:把数据层填上代码,主要工作量是自己写「批量拉多股票、长历史数据」的逻辑。
