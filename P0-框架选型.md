## 0. 一句话总览（先看这个）

**3 个候选框架全部是「LLM 多 Agent 投研决策系统」，没有一个是传统量化 ML 框架。**
3 个 repo 里**都没有** LightGBM / LSTM / sklearn / torch —— 它们靠 LLM 辩论出买卖决策，不靠机器学习模型预测。

这意味着：我们计划里的「传统量化核心」（量价因子 → LightGBM/LSTM → 4 类预测目标 → 回测）**这 3 个框架都没有现成的，需要自己写**。它们能复用的是「外围基础设施」：A股 数据层、因子计算的零件、回测引擎、LLM 调用封装（Stage 2 LLM 因子会用到）。

所以选型的真正问题不是「选哪个当主框架」，而是「**从这 3 个里各拆哪些零件，拼出我们自己的混合架构**」。详见第 4 节。

## 1. 逐框架分析

### 1.1 TradingAgents-astock（simonlin1212）

- **仓库**：https://github.com/simonlin1212/TradingAgents-astock
- **许可证**：Apache 2.0（可自由使用 / 二次开发 / 商用，保留 NOTICE 即可）—— 对我们最友好
- **最近更新**：2026-05-14（一天前），版本 0.2.4
- **定位**：TauricResearch/TradingAgents 的 **A股 深度特化 fork**。本质是「LLM 多 Agent 投研框架」——7 个分析师各写研报 → 多空辩论 → 风险三方辩论 → Portfolio Manager 出 Buy/Hold/Sell + 仓位。**不是预测引擎，是「AI 投研团队」模拟器**。

**A股 适配度：★★★★★（最高）**
这是 3 个框架里唯一**原生为 A股 改造**的：
- 数据源全换成 A股 免费源：`mootdx`（TCP 行情/财务/F10）+ 腾讯财经（PE/PB/市值）+ `akshare`（财报三表 sina 源 / 新闻 / 一致预期）
- 交易规则按 A股 来：T+1、涨跌停、最小手数、ST、交易时段
- 基准换成沪深300（不是 SPY）
- 中文报告输出
- 3 个 A股 特色分析师：政策分析师、游资追踪师、解禁监控师

**预测能力：无 ML。** 全靠 LLM。没有因子库、没有模型训练。

**回测能力：实质为零（重要坑）。**
`pyproject.toml` 里声明了 `backtrader>=1.9` 依赖，但**全仓库没有任何一行 `import backtrader`** —— 这个依赖是摆设。它的 `graph/reflection.py` 只做一件事：一笔决策跑完后，用 yfinance 拉 5 日后的价格，算「5日收益 vs 沪深300」写进记忆日志。这是「单次决策的事后复盘」，**不是回测引擎**——没有逐日推进、没有持仓管理、没有 Sharpe/回撤/净值曲线。**想做回测得自己写或从别处搬。**

**与「混合架构」的契合度：**
- ✅ **做传统量化核心**：它的 `dataflows/a_stock.py`（~750 行，17 个数据方法）是**整个选型里最值钱的资产**——把 a-stock-data 那些零散 Python 片段封装成了一个**干净、可 import、带 CSV 缓存、带 look-ahead 防护**的模块。`_load_ohlcv_astock()` 直接给你「mootdx 拉 800 根日线 → 缓存到 CSV → 按 curr_date 截断防未来函数」。P2 数据管道可以直接拿来用 / 改。它的 `get_indicators()` 用 `stockstats` 算 RSI/MACD/BOLL/ATR 等 —— 这是现成的**量价因子计算零件**。
- ✅ **做 LLM 因子（Stage 2）**：它的 7 个分析师 prompt（尤其政策/游资/解禁/舆情）+ `llm_clients/`（多供应商封装：OpenAI/Anthropic/Google/DeepSeek/Qwen/GLM/MiniMax/Kimi）是 Stage 2「LLM 把新闻转因子」的高质量参考甚至可直接复用。它的 `agents/utils/structured.py` 做 LLM 结构化输出，正好是「LLM 吐出结构化因子值」需要的。
- ⚠️ 但它的核心——LangGraph 多 Agent 辩论拓扑（`graph/setup.py` / `conditional_logic.py`）——**和我们的混合架构正交**，我们不需要「让 AI 辩论出决策」，我们要「LLM 出因子 → 喂给 ML 模型」。这部分代码读它学思路即可，不直接搬。

**学习友好度：★★★★☆**
代码质量高、注释充分、模块边界清晰。文档极全：`README.md` + `CHANGES_FROM_UPSTREAM.md`（21KB，逐 Week 记录改了什么 + 踩坑）+ `DEV_LOG.md`（33KB）。`examples/cases/` 有比亚迪/宁德/茅台的真实分析输出可参考。对「学习用」非常友好。唯一门槛：要跑通完整流程需要 LLM API Key（每次分析 30-50 次 LLM 调用），但我们主要是拆零件不是跑全流程。

**工程健康度：** Apache 2.0、一周内有更新、有 `tests/` 目录、有 E2E 测试（`test_astock.py`）。个人项目（作者 Simon 林），star 数不高（fork 项目），但代码成熟度和维护活跃度都不错。

### 1.2 ai-hedge-fund（virattt）

- **仓库**：https://github.com/virattt/ai-hedge-fund
- **许可证**：MIT（最宽松，可自由使用 / 商用 / 闭源二改）
- **最近更新**：2026-05-09（几天前）
- **定位**：知名开源项目（virattt，Twitter 大V，star 数最高的一个）。**「AI 对冲基金」**——把 13 个知名投资人（巴菲特、芒格、Cathie Wood、Damodaran…）做成 LLM persona agent，加 4 个分析 agent（估值/情绪/基本面/技术）+ 风险经理 + 组合经理。**同样是 LLM 决策系统，不是 ML 预测**。

这个 repo 有**两套代码**，要分开看：

#### v1（`src/` 目录）—— 完整、可跑、但面向美股
- **LLM persona agents**：每个 agent 内部其实是「用 Python 算一堆规则信号 + 让 LLM 按某投资人风格做判断」。比如 `src/agents/technicals.py` 是**纯 Python 算了 trend/mean-reversion/momentum/volatility/统计套利 5 类信号**——这部分**就是现成的量价因子计算代码**，逻辑清晰、可直接抽出来当因子库的种子。`fundamentals.py` 同理算基本面信号。
- **v1 回测器（`src/backtesting/`，~1000 行，11 个文件，完整可用）**：这是**整个选型里唯一一个完整、能跑的回测引擎**。`BacktestEngine` 逐日推进、`Portfolio` 管持仓和现金、`PerformanceMetricsCalculator` 算 Sharpe / Sortino / 最大回撤、`BenchmarkCalculator` 对基准、支持做多做空和保证金。**这是 P4 回测阶段最值得搬的东西。**
- **数据源**：`FINANCIAL_DATASETS_API_KEY` —— 美股付费数据 API，**A股 完全不可用**。

#### v2（`v2/` 目录）—— 设计很好，但是半成品
README 第一行明说 `Status: Work in Progress, not yet integrated`。这是作者「推倒重来、用规范的量化管线替换 persona agents」的尝试，方向**和我们的传统量化核心高度一致**：
```
Data → Signals → Features → Portfolio Construction → Risk → Execution
```
- ✅ **设计层面值得抄**：`v2/signals/base.py` 的 `BaseSignal` ABC（每个信号输出 `[-1,+1]`，纯 Python 无 LLM）、`v2/models.py` 的 Pydantic 数据契约（`SignalResult` / `QuantSignals` / `PortfolioTarget`）、`v2/data/protocol.py` 用 `Protocol` 做数据源抽象（任何实现这几个方法的类都能当数据源——**正好可以让 a-stock-data 适配进来**）。它的设计原则也对路：「成本从第一天就建模」「CPCV + PBO 防过拟合是一等公民」「point-in-time 防未来函数」。
- ✅ **`v2/event_study/` 是唯一完整实现的模块**（~900 行）：事件研究框架，算财报公告前后的累计异常收益 CAR、市场模型 OLS、t 检验、bootstrap 置信区间。这套**事件研究方法论可以直接迁移到 A股**做「财报/解禁/龙虎榜事件因子」。
- ❌ **但 backtesting / features / portfolio / risk / pipeline / validation 全是空壳**——每个文件只有 1-5 行 docstring，没有实现。CPCV、PBO、向量化回测器、Almgren-Chriss 市场冲击模型……都只是 README 里的承诺，代码没写。

**A股 适配度：★☆☆☆☆**
v1 数据层绑定美股付费 API，v2 同样只支持美股。整个 repo **没有任何 A股 适配**。要用必须自己接数据层。

**预测能力：无 ML。** （v2 的 signals 也是纯 Python 数学，不是 ML 模型，但比 v1 的 persona agent 更接近「因子」的形态。）

**回测能力：v1 完整可用 ★★★★☆，v2 是空壳 ★☆☆☆☆。**

**与「混合架构」的契合度：**
- ✅ **做传统量化核心**：v1 回测器（搬过来换数据接口即可）、v1 `technicals.py`/`fundamentals.py` 的因子计算逻辑（抽出来当因子库种子）、v2 的整套**架构设计和数据契约**（`BaseSignal` ABC、Pydantic models、`Protocol` 数据抽象——这些是「怎么组织一个干净量化项目」的最佳范本）、v2 `event_study/` 的事件研究方法论。
- ✅ **做 LLM 因子**：v1 有 `news_sentiment.py` / `sentiment.py`，是 LLM 做情绪的参考。
- ⚠️ 注意 v2 是 WIP，**不能指望它开箱即用**，价值在「设计蓝图」不在「成品代码」。

**学习友好度：★★★☆☆**
v1 代码可读，但 persona agent 那套（13 个投资人）对我们的目标是噪音。v2 设计文档（`v2/README.md`）写得很好，**很适合 architect 在 P1 设计项目结构时参考**。但代码注释比 TradingAgents-astock 少，`black` 行宽设成了 420（极不常规，可读性打折）。

**工程健康度：** MIT、star 数最高、有 `.github/`、有 `tests/`、几天前更新。但 `pyproject.toml` 里 `authors = ["Your Name <your.email@example.com>"]` 没填——细节上不够严谨。

## 2. 数据 skill 能力摸底：a-stock-data

- **仓库**：https://github.com/simonlin1212/a-stock-data
- **许可证**：Apache 2.0
- **最近更新**：2026-05-12，版本 2.1
- **作者**：simonlin1212 —— **和 TradingAgents-astock 同一个作者**（这点很关键，见下）

### 2.1 它是什么形态？

**它是一个 Claude Code Skill，不是一个 Python 库。** 仓库主体是单个 `SKILL.md`（1515 行）= 结构化 Markdown + 内嵌 Python 代码片段。使用方式有两种：
1. **作为 Skill**：丢进 `~/.claude/skills/a-stock-data/SKILL.md`，Claude Code 在 A股 对话里自动激活，按需执行内嵌代码。
2. **当代码片段库**：把里面的 Python 函数复制到你自己的脚本里跑（README 明确支持这种用法）。

⚠️ **重要**：它**不能 `pip install` 然后 `import`**。它没有包结构，是「给 AI 读的代码手册」。但是——**TradingAgents-astock 的 `dataflows/a_stock.py` 已经把它封装成了真正可 import 的模块**（作者自己做的，因为是同一个人）。所以 P2 数据管道有两条路：(a) 直接抄 SKILL.md 里的片段，(b) 直接用 TradingAgents-astock 的 `a_stock.py`。**(b) 更省事**——已经做好了缓存、ticker 归一化、look-ahead 防护、错误处理。

### 2.2 它提供什么数据？（6 层 21 端点，8 个数据源）

| 层 | 端点 | 数据 | 实时/历史 |
|---|---|---|---|
| **行情层** | mootdx | K线（日/周/月/分钟，`offset` 控制根数）+ 五档盘口 + 逐笔成交 + 46字段实时报价 | **两者都有**：K线可拉历史（`offset=800`≈3年日线），盘口/逐笔是实时 |
| | 腾讯财经 | PE(TTM) / PE(静) / PB / 总市值 / 流通市值 / 换手率 / 涨跌停价 | 实时快照 |
| **研报层** | 东财 reportapi | 研报列表 + 评级 + 三年 EPS 预测 + PDF 下载 | 历史可查 |
| | akshare 一致预期 | 同花顺机构一致预期 EPS（按年度） | 当前快照 |
| | iwencai NL 搜索 | 自然语言研报检索（**唯一需要 API Key 的端点**） | — |
| **信号层** | 同花顺热点 | 当日强势股 + **题材归因 reason tags（人工标注）** | 实时（当日，历史拉不到）|
| | 同花顺北向 | 沪股通/深股通分钟级流向；历史改本地自缓存 | 实时 + 自缓存历史 |
| | 百度概念板块 | 行业/概念/地域三维归属 | 实时 |
| | 百度资金流向 | 主力/散户/超大单/大单分钟级 + **20日历史** | 实时 + 20日历史 |
| | 龙虎榜席位 | 上榜记录 + 买卖席位 TOP5 + 机构动向 | **历史可查** |
| | 全市场龙虎榜 | 每日全市场上榜 + 净买额排名 | **历史可查** |
| | 限售解禁日历 | 历史解禁批次 + 未来90天待解禁 | **历史 + 未来** |
| | 行业横向对比 | 同花顺~90行业涨跌排名 + 资金流 + 领涨股 | 实时（当日）|
| **新闻层** | akshare ×3 | 个股新闻（东财）/ 财联社快讯 / 全球资讯 | 实时流（近期）|
| **基础数据** | mootdx finance | **37字段季报**（EPS/ROE/净利润/主营收入/每股净资产…）| 历史季报 |
| | mootdx F10 | 9大类公司文本资料 | 当前快照 |
| | akshare 个股基本面 | 总股本/流通股/总市值/行业/上市时间 | 当前快照 |
| **公告层** | 巨潮 cninfo | 沪深北全量公告 | 历史可查 |

**字段覆盖总结**：行情 ✅、财务 ✅（季报37字段 + 财报三表经 akshare sina 源）、资金流 ✅（北向/主力/龙虎榜）、新闻公告 ✅、研报情绪 ✅。**对「量价因子 + 财务因子 + 资金流因子 + LLM 文本因子」四类因子的数据需求基本全覆盖。**

### 2.3 用于「构建历史数据集」的能力评估（关键）

我们要训练 ML 模型，需要的是**多股票 × 长时间的面板数据**。a-stock-data 的设计重心是「**查一只股票的当前状态**」（估值/题材/资金流），不是「批量下载历史 panel」。具体说：
- ✅ **能拿到训练所需的历史数据**：mootdx K线 `offset` 可拉~800 根日线（≈3年），财务季报有历史，龙虎榜/解禁有历史。TradingAgents-astock 的 `_load_ohlcv_astock` 已经实现了「拉 + 缓存 CSV」。
- ⚠️ **但没有「批量 universe + 增量更新」的现成能力**：要构建「全市场 N 只票 × 5 年」的数据集，得自己写循环 + 落地 + 增量逻辑。这是 **P2 data-engineer 的主要工作量**。
- ⚠️ **部分信号层端点只有当日 / 短历史**（同花顺热点、行业对比当日；北向、百度资金流只有 20 日 / 自缓存）——这些做**实时信号**很好，做**长历史回测的因子**会缺数据。需要 P2 评估：要么靠每日跑积累自缓存，要么这些因子只用于近期、不进长周期回测。
- ⚠️ **mootdx 走 TCP 直连通达信服务器，需要国内 IP**。海外/代理环境可能超时（README 明确提到）。

### 2.4 调用方式小结

P2 复用建议：**直接用 TradingAgents-astock 的 `tradingagents/dataflows/a_stock.py`** 作为数据层起点（它就是 a-stock-data 的工程化封装，同作者），需要 a-stock-data SKILL.md 没被封装进去的端点（如 iwencai 语义搜索）时再回去抄 SKILL.md 的片段。依赖：`pip install mootdx akshare requests pandas stockstats`。

## 4. 综合选型建议

### 4.1 核心判断

不存在「选一个当主框架就行」的方案——因为**没有任何一个框架提供我们要的传统量化 ML 核心**。正确做法是「**主数据/LLM 框架 + 回测/架构补充库 + 自建 ML 层**」的组合。下面给**一个推荐方案 + 一个备选**。

### 4.2 推荐方案（方案 A）：TradingAgents-astock 打底 + ai-hedge-fund 补回测/设计 + 自建 ML

| 组件 | 选谁 | 承担角色 | 理由 | 用于 |
|---|---|---|---|---|
| **数据层** | TradingAgents-astock `dataflows/a_stock.py`（背后是 a-stock-data）| A股 数据接入：行情/财务/新闻/资金流/龙虎榜/解禁 | 唯一原生 A股、已工程化封装、带缓存和 look-ahead 防护、同作者保证一致性 | **Stage 1 P2** |
| **量价因子库 种子** | TradingAgents-astock `get_indicators`（stockstats）+ ai-hedge-fund v1 `technicals.py` 的因子逻辑 | 提供技术指标/动量/均值回归/波动率因子的计算代码起点 | 两边都是现成可抽取的因子计算代码，省去从零写指标 | **Stage 1 P3** |
| **项目架构 / 数据契约** | ai-hedge-fund **v2 的设计**（`BaseSignal` ABC、Pydantic models、`Protocol` 数据抽象、目录分层）| 给 architect 在 P1 当「干净量化项目长什么样」的蓝图 | v2 的 `Data→Signals→Features→Portfolio→Risk` 分层和我们的混合架构几乎同构，且强调成本建模/防过拟合/point-in-time | **Stage 1 P1** |
| **ML 模型层** | **自建**（LightGBM / LSTM）| 4 类预测目标的核心预测引擎 | 3 个框架都没有，必须自己写。这恰好是「学习与研究」项目最有学习价值的部分 | **Stage 1 P3** |
| **回测引擎** | ai-hedge-fund **v1 `src/backtesting/`** 搬运改造 | 逐日回测 + 持仓管理 + Sharpe/Sortino/回撤 + 基准对比 + 信号产出 | 选型里唯一完整可跑的回测器；换掉美股数据接口、加上 A股 T+1/涨跌停约束即可 | **Stage 1 P4** |
| **事件研究方法** | ai-hedge-fund **v2 `event_study/`** 迁移 | 财报/解禁/龙虎榜「事件因子」的方法论参考 | 唯一完整实现的事件研究框架（CAR/市场模型/显著性检验），方法论可直接迁 A股 | Stage 1 P3 因子 / Stage 2 |
| **LLM 调用封装** | TradingAgents-astock `llm_clients/` + `agents/utils/structured.py` | Stage 2 调 LLM、让 LLM 吐结构化因子 | 多供应商（含国产 + Kimi）已封装好，结构化输出现成 | **Stage 2 P6** |
| **LLM 因子 prompt 参考** | TradingAgents-astock 的 7 个分析师 prompt（政策/游资/解禁/舆情/新闻）| Stage 2「LLM 把新闻/研报/公告转情绪+事件因子」的 prompt 蓝本 | 已经是 A股 语境、中文、覆盖我们要的文本因子类型 | **Stage 2 P6** |

**这个方案的逻辑**：A股 数据和 LLM 部分用 TradingAgents-astock（A股 适配最好、同作者生态一致）；回测和量化项目工程范式用 ai-hedge-fund（v1 回测器是唯一成品、v2 设计是最佳范本）；最核心的 ML 预测层自建（无现成 + 学习价值最高）。TradingAgents 原版**不进 Stage 1**，留作上游参考。

**Stage 1 用到**：TradingAgents-astock 数据层 + 因子零件 + ai-hedge-fund v1 回测器 + v2 架构设计 + 自建 ML。
**Stage 2 用到**：TradingAgents-astock 的 `llm_clients/` + 分析师 prompt + structured 输出，把 LLM 情绪/事件因子接进 P3 因子层。

### 4.3 备选方案（方案 B）：最小依赖自建，框架只当「代码参考」不当「依赖」

如果用户更看重「项目纯净、可读、每一行都自己掌控」（学习型项目的合理偏好），可以**不把任何框架作为 pip 依赖**，而是：
- 自己起一个干净的项目骨架（architect 参考 ai-hedge-fund v2 的设计）
- 数据层：照抄 a-stock-data SKILL.md 的片段 + TradingAgents-astock `a_stock.py` 的封装思路，自己写一份精简版（只留我们要的端点）
- 因子库、ML 层、回测器全部自建，但**回测器重度参考 ai-hedge-fund v1**、因子参考两个 repo 的指标代码
- 好处：没有 LangGraph / langchain / backtrader 一大堆重依赖，项目轻、概念少、最适合「读懂每一行」
- 代价：前期工作量更大，回测器要从参考代码重写而不是直接搬

**方案 A vs B 的权衡**：
- **方案 A** 上手快、能复用成熟代码（尤其 v1 回测器和 a_stock.py 直接搬），但会引入 langchain 生态的重依赖，项目体积和概念负担较大。
- **方案 B** 项目最干净、学习路径最纯粹、完全可控，但 P2/P4 自己写的活更多。
- 我个人倾向 **方案 A 但带「瘦身」纪律**：依赖只装真正用到的（数据层那几个包是必须的；LangGraph 在 Stage 1 其实用不到，可以 Stage 2 再引入）。具体取舍建议交 **architect 在 P1 拍板**——它设计目录结构时自然会决定「搬代码」还是「连依赖」。

### 4.4 给用户的待决问题

1. **方案 A 还是方案 B？**（复用成熟代码省事 vs 项目纯净可控）—— 也可以让 architect 在 P1 给出更细的目录方案后再定。
2. **4 类预测目标的深度**（计划里 P0 后要定的悬而未决项）：基于现在的认知——既然 ML 层完全自建、ai-hedge-fund v1 回测器对「单资产时序回测」和「横截面组合」都支持，建议 **Stage 1 先把「① 涨跌方向二分类」一条链做透**（数据→因子→LightGBM→回测→信号全跑通），② 回归 / ③ 选股排序 / ④ 买卖信号作为「复用同一套基础设施的扩展点」，P3 模型阶段视进度逐个补齐。这样验收标准（计划里的 Stage 1 验收 ①-⑤）能最快达成。
3. **mootdx 的国内 IP 依赖**：当前运行环境是否在国内 / 能否直连通达信服务器？如果在海外，P2 数据层要提前规划代理或备用源——这会影响 P2 的方案。
