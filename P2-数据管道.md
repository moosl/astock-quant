## 0. 一句话结论

`astock_quant/data/` 四个文件全部填实，`contracts.py` 的 4 个数据层契约补全，`config/settings.py` 落定 Stage 1 起步配置。**实测用 30 只蓝筹拉了真实 A股 数据**：行情 panel、资金流 panel、财务字典都能构建，CSV 缓存生效，look-ahead 截断验证通过。下游 factors / labels 层现在有真实数据可消费。

## 1. 建了什么

### 1.1 `contracts.py` —— 4 个数据层契约（只动这 4 个）

| 契约 | 关键字段 | 说明 |
|---|---|---|
| `PriceBar` | ticker, date, OHLC, volume, amount | 单根日线。`amount`（成交额，元）允许 None。 |
| `FinancialMetrics` | ticker, report_period, eps, roe, revenue, net_profit, net_assets, bvps, pe, pb, total_share, float_share | 单只票某报告期财务。除 ticker/report_period 外**全部允许 None** —— 数据源拼不齐很正常。 |
| `MoneyFlowRecord` | ticker, date, main_inflow, super_inflow, large_inflow, northbound, dragon_tiger_net | 单只票某日资金流。**除 ticker/date 外全可缺** —— P0 已警告这类端点历史短。 |
| `NewsItem` | ticker, date, title, content, source, url | 单条新闻。Stage 1 固定形状，Stage 2 LLM 因子消费。`ticker=None` 表示大盘新闻。 |

其它契约（FactorValue / FactorFrame / Label / Prediction / BacktestResult / SignalReport）保持 stub，留给后续阶段。

### 1.2 `config/settings.py` —— Stage 1 起步配置

- **UNIVERSE**：30 只流动性好的大盘蓝筹，跨 12+ 行业（白酒/银行/非银/新能源/医药/家电/科技/资源/基建/汽车…），沪深两市都覆盖。这是**起步池，不是最终池** —— 要扩到沪深300 全量直接换 `STAGE1_UNIVERSE` 这个 list，数据层对池子大小无感。
- **历史区间**：`history_start=2022-01-01` ~ `history_end=2026-05-15`，约 4.4 年，够 ML 训练 + 时序切分 + 因子预热。
- **路径**：`data_cache_dir` / `artifacts_dir` 以本文件位置推导项目根（不依赖运行时 cwd），都在 `.gitignore` 内。
- 另含 `BacktestConfig` / `FactorWindows` / `SplitConfig` / `LabelConfig` 占位（给 P3/P4 合理默认值，本阶段不深究）。
- 用法：`from astock_quant.config.settings import SETTINGS`，全局单例。

### 1.3 `data/` 四个文件

| 文件 | 职责 |
|---|---|
| `protocol.py` | `DataSource` Protocol（`typing.Protocol`，结构化类型，无需继承）。4 个方法：`get_prices` / `get_financials` / `get_moneyflow` / `get_news`。约定：失败返回空 list 不抛异常，返回 Pydantic 对象不返回裸 DataFrame。 |
| `astock_source.py` | `AStockSource` —— 实现 Protocol。研读 `a_stock.py` 后用自己的话重写的精简版，**未 import 那个框架**。 |
| `cache.py` | CSV 缓存 + **look-ahead 截断**。核心是 `truncate_by_date` —— 所有「按 curr_date 取数据」的地方都过这道闸。 |
| `dataset.py` | **本阶段主要工作量** —— universe 循环 + 拼 panel + 缓存判断 + 增量更新。 |

## 2. 关键决策

### 2.1 数据源分工

研读 `a_stock.py` 后，按「量化管道实际要什么」精简，只留 4 类原始数据的端点：

| Protocol 方法 | 数据源 | 备注 |
|---|---|---|
| `get_prices` | mootdx (TCP 7709) 日线 | 单次最多 800 根，**用 `start` 参数往回翻页**拼接到 `history_start` |
| `get_financials` | akshare `stock_financial_abstract`（EPS/ROE/营收/净利/净资产/BVPS）+ mootdx `finance`（总股本/流通股本）+ 腾讯财经（实时 PE/PB） | 三源拼一条记录 |
| `get_moneyflow` | akshare `stock_individual_fund_flow`（主力/超大/大单净流入） | 历史短，正常 |
| `get_news` | akshare `stock_news_em`（个股新闻，东财源） | Stage 1 就实现，Stage 2 才消费 |

**为什么财务不用 mootdx 的 `finance` 当主源**：实测当前 mootdx 版本的 `finance` 返回的是 37 个**原始资产负债表科目**（`jinglirun`/`zhuyingshouru`…），没有直接的 `eps`/`roe` 字段（SKILL.md 写的版本和实际装的版本字段不一致）。`ak.stock_financial_abstract` 是规整的「指标 × 报告期」宽表，直接有 `基本每股收益`/`净资产收益率(ROE)`，所以财务主源用它，mootdx 只取股本快照。

### 2.2 防未来函数（look-ahead bias）—— 头号大敌

两点设计：

1. **`cache.truncate_by_date`** 是第一道防线，独立成工具函数：给定 `curr_date`，只保留 `date <= curr_date` 的行，严格大于的一律丢弃。**不只行情** —— 财务、资金流、新闻所有时点查询都过它。
2. **缓存存「全量历史」，截断在读取时做**。CSV 里不截断（这样查不同区间不用反复拉网络），`build_*_panel(curr_date=X)` 给定 X 时每只票读缓存都过 `truncate_by_date`。`curr_date=None` 返回全量 panel（仅给「一次性切训练/验证集」这种离线用途；**逐日回测必须传 `curr_date`**）。
3. 财务数据的截断在 `astock_source.get_financials` 内部就做了 —— 报告期晚于 `end_date` 的列直接不返回。

这是与 `models/splits.py` 的 purge gap 配套的**第一道防线**。

### 2.3 缓存 + 增量更新策略

- 每只票的行情/资金流各存一份「全量历史」CSV（`data_cache/{ticker}-prices.csv` 等）。
- 构建 panel 时：缓存**今天更新过** → 直接读 CSV 不打网络；**过期/不存在** → 走 `astock_source` 重拉、落地、再读。
- 「增量」体现在「按天有效」：A股 日线一天一更，当天内重复构建 panel 复用缓存，跨天自动重拉。
- **刻意没做**「只追加尾部 N 天」那种精细增量 —— 实测重拉 30 只票约 30s，不值当为这点时间引入复杂度。换 universe / 怀疑数据有问题时用 `force_refresh=True` 全量重拉。

### 2.4 财务数据为什么不做 panel

行情/资金流是规整日频 → 做成 `MultiIndex=(date, ticker)` 的 panel。财务是季度粒度、按报告期不规则发布，做 panel 反而别扭。所以 `load_financials` 返回 `{ticker: list[FinancialMetrics]}`，由 fundamental 因子层负责按交易日 forward-fill 对齐（某交易日用「最近一期已发布财报」）。

## 3. 怎么用这个数据层

```python
from astock_quant.data.dataset import (
    prepare_stage1_data,    # 一键准备：行情 panel + 资金流 panel + 财务字典
    build_price_panel,      # 单独构建行情 panel
    build_moneyflow_panel,  # 单独构建资金流 panel
)

# —— 最省事：一键准备 Stage 1 全部数据（走 SETTINGS 默认 universe + 区间）——
data = prepare_stage1_data()
prices = data["prices"]          # DataFrame, MultiIndex=(date, ticker), 列=OHLCV+amount
moneyflow = data["moneyflow"]    # DataFrame, 同结构
financials = data["financials"]  # {ticker: list[FinancialMetrics]}

# —— 逐日回测场景：必须传 curr_date 做 look-ahead 截断 ——
panel_t = build_price_panel(curr_date="2024-06-30")  # 只含 <= 2024-06-30 的行

# —— 自定义 universe / 区间 ——
panel = build_price_panel(
    universe=["600519", "000858"],
    start_date="2023-01-01",
    end_date="2025-12-31",
)

# —— 怀疑缓存有问题，强制全量重拉 ——
data = prepare_stage1_data(force_refresh=True)
```

直接用数据源（不经 panel）：

```python
from astock_quant.data.astock_source import AStockSource
src = AStockSource()
bars = src.get_prices("600519", "2022-01-01", "2026-05-15")   # list[PriceBar]
fins = src.get_financials("600519", "2026-05-15")              # list[FinancialMetrics]
```

## 4. 实测结果

环境：本机直连（mootdx TCP 通，akshare/腾讯 HTTP 通），2026-05-15 实跑，30 只起步 universe。

**① 全量构建（`prepare_stage1_data(force_refresh=True)`，强制拉网络）—— 耗时 32.3s**

| 产物 | 结果 |
|---|---|
| 行情 panel | `(31603, 6)`，MultiIndex=(date, ticker)，30 只票全部拿到，日期 2022-01-04 ~ 2026-05-15 |
| 每只票行数 | min=1044 / max=1054 / mean=1053（≈4.3 年日线，分页拼接成功覆盖到 2022 年初）|
| 资金流 panel | `(3600, 5)`，30 只票，日期 2025-11-12 ~ 2026-05-14（**仅 ~6 个月，P0 预期内的短历史**）|
| 财务 | 30/30 只票有数据。示例 600519：102 期，最新 20260331 `eps=21.76 roe=10.57 pe=20.2 pb=6.17` |

**② 缓存生效验证 —— 第二次 `build_price_panel()` 耗时 0.07s**（首次 32.3s → 缓存命中快 ~460 倍），panel 形状一致 `(31603, 6)`，`data_cache/` 下落地 60 个 CSV 文件（30 行情 + 30 资金流）。

**③ look-ahead 截断验证（核心）—— 通过**

- `build_price_panel(curr_date="2024-06-30")` → 截断后 panel `(18023, 6)`，最大日期 **2024-06-28**（≤ 2024-06-30）。
- 对比全量 panel 最大日期 2026-05-15 —— 证明截断确实丢弃了 ~13500 行未来数据，防未来函数防线生效。

结论：**数据层四个文件全部能跑通真实 A股 数据，缓存生效，look-ahead 截断验证通过。**

## 5. 坑 / 给后续阶段的提醒

### 给 P3 因子层

- **资金流 panel 的日期覆盖远短于行情 panel**。akshare 个股资金流只有最近 ~20 个交易日历史（北向个股级更是基本拿不到）。`moneyflow.py` 因子和行情 panel 对齐时会出现**大面积 NaN，这是正常的** —— 因子计算要对缺失鲁棒，不要假设资金流和行情同长。
- **财务数据是 `{ticker: list[FinancialMetrics]}` 不是 panel**。`fundamental.py` 需要自己把报告期记录按交易日 forward-fill 对齐（交易日 T 用「报告期 <= T 的最近一期」）。`get_financials` 已保证不返回未来报告期，但因子层对齐时仍要注意：财报有「发布滞后」（如年报 3-4 月才发），严格说应该用「财报实际发布日」而不是「报告期末日」对齐 —— Stage 1 可以先用报告期末日近似，但要在因子里留个 TODO 注释。
- `FinancialMetrics` 很多字段会是 None（akshare 对老报告期没有 EPS/ROE；总股本/PE/PB 只挂在最新一期上避免误用历史估值）。因子计算前先 dropna 或填充。
- 量价因子直接吃 `build_price_panel()` 的 panel，`MultiIndex=(date, ticker)`，列是 `open/high/low/close/volume/amount`。`stockstats` 算指标时注意它要的列名是小写 `open/close/...`，正好对得上。

### 给 P4 回测层

- **逐日回测必须给 `build_price_panel` 传 `curr_date`**，否则 panel 会包含未来数据，look-ahead 防线失效。`curr_date=None` 只用于离线一次性切分。

### 数据源本身的坑

- mootdx `bars` 的 `offset` 参数实测**封顶 800**（再大也只返回 800 根）。要更长历史必须用 `start` 参数往回翻页 —— `astock_source.get_prices` 已实现分页（`start += 800` 循环直到覆盖起始日期）。
- mootdx `bars` 返回的 DataFrame **index 名是 `datetime`，同时还有一个同名 `datetime` 列**外加 `year/month/day/hour/minute` 拆解列 —— `reset_index()` 前必须先 drop 重名列，否则报错。`get_prices` 已处理。
- mootdx 的 `vol` 和 `volume` 列实测**值一致**（已是统一口径），`amount` 单位是元。
- 当前装的 mootdx 版本 `finance()` 返回 37 个原始科目字段，**没有** SKILL.md 里写的 `eps/roe/profit/income` —— 别照 SKILL.md 的字段名取，会取空。
- 腾讯财经字段索引：**39=PE(TTM)，44=总市值(亿)，46=PB，38=换手率%**。网上很多教程把 43 当 PB 是错的（43 是振幅%）。
- 若在海外环境跑 mootdx TCP 超时：用户有 VPN，非阻塞项；真连不上需切代理或换数据源。

### 落地纪律

- `_p2_verify.py` 是验收脚本，跑完即删，不是项目入口。
- `data_cache/` 下的 CSV 已 gitignore，不进版本控制。

## 补丁记录 · 2026-05-15

### 根因

P3a 审核（第 7 节）发现：`read_cache` 用 `pd.read_csv(...)` 时 pandas 会把全数字列自动推断为 `int64`，导致 `'000858'` 读出来变成 `858`，前导零丢失。下游 factor-engineer 被迫在 `_build_ticker_map` 加 `lstrip("0")` 兜底。

### 修改内容（`cache.py`）

1. 在 `read_cache` 的 `pd.read_csv(...)` 加 `dtype={"ticker": str}`，关掉整数推断。
2. 读取后再过 `_coerce_ticker_column()`，用 `.astype(str).str.zfill(6)` 补回前导零（兜底历史旧缓存）。
3. `write_cache` 写入前同样过 `_coerce_ticker_column()`，保证写入侧 ticker 也是字符串。
4. 在模块顶部加 `_TICKER_COL` 常量 + `_coerce_ticker_column` helper，配套 docstring 说明根因。

### 自测结果

- 写入 `ticker='000858'` → 读出 `['000858', '000858', '000858']`，dtype=str，前导零完整。
- `prepare_stage1_data()` 跑全量 30 只票，panel shape=(31603, 6)，ticker index level 全部是 6 位字符串，无 int 串混入。
