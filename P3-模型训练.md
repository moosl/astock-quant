## 1. 建了什么

### 1.1 `contracts.py` —— 补全 Label / Prediction 两个契约

只动这 2 个（数据层 4 个 P2 已填、因子层 2 个 P3a 已填，其余 P4 BacktestResult / SignalReport
保持 stub）：

| 契约 | 字段 | 说明 |
|---|---|---|
| `Label` | ticker, date, target_type, value | T 日观察、看未来 N 日表现的训练标签。`target_type` ∈ `direction/return/ranking/trade_signal`。Stage 1 只产 direction：value 是 0.0 / 1.0。 |
| `Prediction` | ticker, date, target_type, value, score?, proba? | 模型输出。direction 时 value = 硬分类（0/1），score = P(涨) ∈ [0,1]，proba = (P跌, P涨)。 |

`Prediction` 把硬分类 + 连续分数 + 概率元组**三者都带**：value 给「需要离散答案」的回测、
score 给「不同阈值实验 / 选 Top N」、proba 给「显式校准曲线」。下游按 target_type 分派时
对所有 4 类目标都用同一个契约 —— 这就是 P1 「4 类目标共用基础设施」的落点之一。

### 1.2 `labels/targets.py` —— direction 做透，②③④ stub

| 函数 | 状态 | 说明 |
|---|---|---|
| `direction_label(price_panel, *, horizon, threshold, for_training, close_col)` | ✅ 做透 | ① 未来 N 日累计收益 > 阈值 → 1，否则 0。MultiIndex=(date, ticker) 的 Series。 |
| `series_to_labels(series, target_type)` | ✅ 工具 | 把 Series 转 list[Label] 契约对象（过滤 NaN）。 |
| `align_xy(factor_data, label_series, ...)` | ✅ 工具 | FactorFrame.data 与 label series 按 (date, ticker) 内连接成 (X, y)。 |
| `return_label / ranking_label / trade_signal_label` | ⏸ stub | 类骨架 + docstring 写明接入方式，`raise NotImplementedError`。 |

### 1.3 `models/` —— 5 个文件

| 文件 | 状态 | 说明 |
|---|---|---|
| `base.py` | ✅ 完整 ABC | `BasePredictor`：`fit(X,y,**)` / `predict(X) -> list[Prediction]` / `save(path)` / `load(path) -> self`。注释约定 `X.index` 必须是 (date, ticker) MultiIndex —— 让 4 类模型共面相同形态。 |
| `direction.py` | ✅ 做透 | `DirectionModel(BasePredictor)`，LightGBM 二分类。fit / predict / save / load 全打通，含 early stopping、feature importance、`predict_score_frame` 高效路径。 |
| `splits.py` | ✅ 做透（命门） | `time_series_split(index, train_end, valid_end, purge_gap_days, label_horizon)` + `TimeSeriesSplit` dataclass。**`gap_days < horizon` 直接 raise ValueError** —— 第二道防 look-ahead。 |
| `ret_regression.py` / `ranking.py` / `trade_signal.py` | ⏸ stub（接口 OK） | 类骨架已存在（继承 `BasePredictor`），`fit/predict/save/load` 均 `raise NotImplementedError`。接口签名跟 P3b 的 BasePredictor 一致，未来接入只填实现。 |

### 1.4 `pipeline/run_direction.py` —— 端到端编排

`run_direction(*, universe, train_end, valid_end, horizon, threshold, purge_gap_days, model_params, ...)` 一个函数把 5 步串起来：

```
data.prepare_stage1_data()             # P2
   ↓
factors.compute_factor_frame()         # P3a → FactorFrame
   ↓
labels.direction_label() + align_xy()  # → (X, y), 去掉 NaN-label
   ↓
models.time_series_split()             # 时序切分 + purge gap（命门）
   ↓
DirectionModel.fit / predict + 评估    # → list[Prediction] + metrics
```

返回 dict：`predictions`, `metrics`, `factor_names`, `feature_importance`, `split`,
`model`, `score_frame`, `y_valid`。

### 1.5 `tests/test_splits_purge.py` —— 新增 11 测试

合成 panel（不依赖 P2 缓存，CI 友好）。覆盖：

- **命门 1：`gap < horizon` 必须 raise**
- **命门 2：train 最大日 与 valid 最小日 之间至少隔 horizon 个交易日**
- 训练 / 验证 不重叠 + 时序顺序正确
- 周末等非交易日自动对齐到前一交易日
- `train_end >= valid_end` raise
- mask 长度正确、`purged_count` = gap × ticker 数
- SETTINGS 默认 `purge_gap >= label.horizon`（永久不变量）

## 3. 关键决策

### 3.1 look-ahead 第二道防线：purge gap（命门）

**问题**：direction label 的语义是「T 日的 y = T+5 日的真实涨跌」。训练 / 验证按 train_end
简单切，训练集尾部 `date ∈ (train_end-5, train_end]` 的样本，其 y 已经看了 train_end+5
日的真实价格 —— 标签泄漏到验证集前段。

**防线**：在切线后挖一段「purge gap」，含至少 `horizon` 个交易日的样本被丢弃。

```
训练集：[start, train_end]
 ──── PURGED (train_end, train_end + gap_days] ────  ← 含未来窗口的样本被丢弃
验证集：(train_end + gap_days, valid_end]
```

**严格执行**：`time_series_split(gap_days < horizon)` 直接 `raise ValueError`。SETTINGS
默认 `purge_gap=10` >= `label.horizon=5`，安全冗余 2 倍；测试用例 `test_default_settings_satisfy_constraint`
守护这个永久不变量。

**实测**：跑 run_direction() 时被 purge 的样本数 = 300（30 票 × 10 个交易日），完全吻合。

### 3.2 gap 按「交易日数」而非「自然日」

自然日有周末 / 假期 / 长假，10 个自然日可能只有 6 个交易日 —— gap 在交易日维度不够。
本模块的 `time_series_split` 从 `index.get_level_values("date")` 取 panel 实际交易日序列，
排序去重后按**位置**跳 `gap_days + 1` 个 step，比加自然日鲁棒。`test_train_valid_gap_satisfies_horizon`
显式核对这点。

### 3.3 label 是唯一允许「负数 shift」的地方

P3a 在 `factors/base.py` 立下纪律：因子层禁用负数 shift。但 label 本质就是「T 时刻看 T+N 的
真实涨跌」，必须用 `pct_change(N).shift(-N)`。**这是 labels 模块唯一允许出现的 forward-looking
操作**，由 splits 的 purge gap 配合守住红线。`direction_label` docstring 显式说明该例外。

`for_training: bool` 参数明示语义：
- `True`（训练）：尾部 horizon 行 label 自然 NaN（pct_change(N).shift(-N) 的副产品），
  中段「看未来」是监督学习的合法用法
- `False`（推理）：实际效果相同 —— 当前 T 之后真实价格未知，尾部 NaN 是天然的「不可知」
  标志。本参数提供清晰的语义钩子让调用方明示意图。

### 3.4 NaN 处理：因子层不填、对齐层删行、模型层原生

P3a 因子层留下的 NaN（资金流 88%、PE/PB 97% 等）经过：
1. `align_xy(drop_label_nan=True, drop_all_nan_rows=True)` —— 删掉 (a) y 为 NaN 的行
   （上下文起始 / 尾部）+ (b) X 全为 NaN 的行（无信息样本）。
2. 剩下的单列 NaN 一路传到 LightGBM —— LightGBM 把 NaN 当独立分支学。

**实测**：31603 行 panel → align_xy 后剩 31453 行（去掉 150 行尾部 NaN）→ 切分训练 25283
+ 验证 5780 + purged 300 + valid_end 之后裁剪 1090 = 32453 与上游不严格相等是因为切分
按 unique 交易日做（每日 ~30 票），数字逻辑自洽。

### 3.5 LightGBM 超参选择

`DirectionModel.DEFAULT_PARAMS`：

```
n_estimators=300, learning_rate=0.05, num_leaves=31, min_child_samples=50,
subsample=0.8 / subsample_freq=1, colsample_bytree=0.8,
reg_alpha=0.0, reg_lambda=0.0, random_state=42
```

设计取向：**Stage 1 以「跑通 + 给合理 baseline」为目标，不做盲目调参**。
- `num_leaves=31`（默认）+ `min_child_samples=50`：防止过拟合，蓝筹股 25 因子量量级合适
- subsample + colsample：行 / 列双随机，进一步抗过拟合
- learning_rate=0.05 + 300 轮 + early_stopping_rounds=30：自动找到最佳轮数

调参留给后续阶段或用户研究 —— 学习型项目重点是「基础设施可用」，不是「这版模型能赚钱」。

### 3.6 BasePredictor 接口对齐 4 类目标

`base.py` 的抽象：`fit(X, y, **kwargs) -> self` / `predict(X) -> list[Prediction]` /
`save(path)` / `load(path) -> self`。约束 `X.index` 必须是 (date, ticker) MultiIndex
—— 这样 ②③④ 模型实现时 predict 拿到 ticker / date 字段没有歧义。`ret_regression.py` /
`ranking.py` / `trade_signal.py` 的 stub 类骨架已存在，签名跟 BasePredictor 一致，
未来填实现时不用动接口。

### 5.6 诚信解读：AUC 0.51 是「跑通基础设施」的产物，不是「能赚钱的模型」

AUC=0.51、Acc=0.53 几乎等于随机。这**不是 bug，是 Stage 1 的诚实 baseline**：

1. **数据规模**：30 只大盘蓝筹 × 5 日涨跌方向 —— 蓝筹股短期方向本就接近随机过程
   （base rate 0.468 在 0.5 附近），用 25 个常规因子拿到 marginal AUC 0.51 是合理的。
2. **无特征工程**：没做横截面归一化、没做行业中性化、没做分位编码 —— 这些一般能多压
   2-5 个百分点。P3a 给的 raw 因子值 LightGBM 直接吃。
3. **无超参调优**：用默认超参跑通即收。一轮 GridSearch 或 Optuna 通常能再涨 1-3 个 pct。
4. **PE/PB 历史几乎全 NaN + 资金流只有 6 个月**：占总因子数 28%（7/25），可学信息被
   阉割。P3a 文档已标 TODO（补齐历史 PE/PB 序列 / 接长历史资金流端点）。
5. **未来 N 日二分类天花板低**：学术界一般认为「单纯涨跌 N 日方向」的可预测 AUC 上限
   在 0.55 ~ 0.58 之间。AUC 0.51 距上限 5-7 个百分点，正常的 baseline。

**Stage 1 验收：跑通端到端 + 守住两道 look-ahead 防线** —— 这两条都做到了。模型本身的
"赚钱能力"是后续阶段的事（特征工程 / 横截面归一化 / LLM 因子 / 调参 / walk-forward）。

## 6. 给后续阶段的提醒

### 给 P4 回测层

- `run_direction()` 返回的 `predictions: list[Prediction]` 或 `score_frame: DataFrame`
  可直接喂回测引擎。**`score_frame` 更高效**：MultiIndex=(date, ticker) 的 DataFrame，
  列 = `['value', 'score', 'proba_down', 'proba_up']`，回测引擎按 date 切片即可。
- **逐日回测必须配合 P2 的 `build_*_panel(curr_date=T)`**：每个交易日重新算因子 +
  调 `DirectionModel.predict()`。P3a 的 panel 截断 + P3b 的 purge gap **只解决训练**
  的 look-ahead，**推理时还是要 P2 截断闸门**。
- `predict(X)` 不需要 `for_training` 标志 —— 因子是「截至 T 的因子」，模型对所有样本一视
  同仁；推理用 T+N 还没发生时该样本就不在 X 里（panel 不包含 T+N 行）。

### 给 P4 信号层 / P5 ② ③ ④

- `BasePredictor` 接口稳定（5 个方法），实现 ②③④ 只需在 `models/*.py` 把 stub 的
  `raise NotImplementedError` 改成实现，**不动 base.py**。
- `labels/targets.py` 的 ②③④ 函数 stub 已留好文档，描述了「复用 direction_label 的
  `pct_change(N).shift(-N)` 上游」的接入方式。
- pipeline 加 `run_return.py / run_ranking.py / run_trade_signal.py`，复用 `run_direction.py`
  的 5 步骨架，只换 labels.* + models.* + metrics（回归用 RMSE/IC，排序用 NDCG/IC，
  trade_signal 用 precision@k）。

### 已知 TODO（Stage 1 不做，但下次迭代可看）

1. **横截面归一化**：训练前对每个 (date, factor) 做横截面 rank/z-score，能让模型更快
   学到「相对强弱」而不是「绝对值」。但要严格在 splits.train 子集 fit、valid 子集 apply，
   避免泄漏 —— 这个细节落到 `preprocessing/` 层（Stage 1 没建，需要时新增）。
2. **类别不平衡处理**：当前 base rate ~0.47（接近平衡）所以无明显问题；如果换成 `threshold=0.02`
   过滤小幅震荡，base rate 会偏离 0.5 较多，需要 `is_unbalance=True` 或 `class_weight`。
3. **walk-forward 验证**：单一切线 + 单一 valid 集只是 baseline。后续可在 `splits.py`
   加 `walk_forward_split(...)` 滚动评估（接口位已经留好）。
4. **早停的实现细节**：当前用 LightGBM 的 callback API，eval_set=valid。但 valid 用于
   早停同时又用于报告 AUC 等指标，严格说有「调参用了验证集」的轻微泄漏 —— 后续可补
   train / valid / test 三段切分（valid 用于早停 + 模型选择，test 用于最终评估）。
