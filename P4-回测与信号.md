## 2. 关键设计决策

### 2.1 look-ahead 命门（回测层的最后防线）

模型层的 prediction 已经标了「这是基于截至 T 的因子做的预测」，回测层的责任是
**不让模型看到 T 之后的数据**。三道防线：

| 数据 | 防线 |
|---|---|
| 当日 bar (close[T]) | mark-to-market + 执行价 |
| 前一日 bar (close[T-1]) | 涨跌停判定（模拟「盘前知道昨收」） |
| 任何 > T 的数据 | 引擎不读取（每日只 `panel.loc[T]` 切片） |

注：Stage 1 用「T 当日 close 成交」近似（与标签 `pct_change(N).shift(-N)` 一致）。
更严谨的做法是「T 收盘出信号 → T+1 开盘成交」，留给后续阶段。

### 2.2 A股 约束 4 件套（写测试守住）

- **T+1**：`bought_today` set 每日清空 + 卖出前查 → 「买入 T 日不能 T 日卖」
- **涨跌停**：主板 10% / 创业板 20% / 科创板 20% / ST 5%；`close >= limit_up - tol` 即视作封板
- **最小手数**：买入 < 100 拒、否则向下取整到百位；卖出可任意（清零头允许）
- **ST 过滤**：默认开（`skip_st=True`），但 Stage 1 没接 ST 名单，st_set 默认空集合

每条都有 pytest 用例（见 `tests/test_constraints_astock.py`），改代码必跑。

### 2.3 交易成本（与 BacktestConfig 默认对齐）

- 佣金：双边 万 3（commission_rate=0.0003）
- 印花税：卖出单边 千 0.5（stamp_tax_rate=0.0005）
- 滑点：5 bps（万 5）—— 买入 close × (1+0.0005)、卖出 close × (1-0.0005)

这套成本对蓝筹偏严、对中小盘偏宽松；高频策略会被它吃掉相当大一块利润，
**这恰好是后面真实回测看到的现象**。

### 2.4 下单策略（Stage 1 等权 Top-K）

- score >= 0.55 → 候选买入（默认 buy_threshold）
- score < 0.45 → 持仓清仓（默认 sell_threshold）
- 同时持仓上限 `max_positions=5`，资金等权分配（cash / slots）
- 没出现在当日 prediction 名单里的票 → 默认清仓（保守，univers 切换友好）

### 2.5 信号语义（4 类目标分派）

- **direction**（Stage 1 做透）：score 决定 buy/sell/hold，strength = |score-0.5|×2，
  reason 写人话（"模型预测涨概率 0.62，≥ 买入阈值 0.55"）
- **return / ranking / trade_signal**：写了 stub 分支，签名稳定，方便后续阶段直接接

## 3. 真实回测结果

### 3.1 默认阈值（buy=0.55 / sell=0.45）—— **0 笔交易**

```
trading_days: 193 (2025-07-15 ~ 2026-04-30)
n_trades: 0
total_return: 0.00%
```

**原因诚实剖析**：P3b 模型 AUC=0.513（已知接近随机），验证集 5780 条预测的 score
基本全部落在 [0.45, 0.55) 这个 hold 区间 —— 模型根本没有「强信号」，自然不下单。

### 3.2 放宽阈值（buy=0.51 / sell=0.49）

```
trading_days: 193 (2025-07-15 ~ 2026-04-30)
n_trades: 14
n_buy_orders: 7
n_sell_orders: 7
total_return: +5.92%
annualized_return: +7.80%
sharpe: 1.105
sortino: 1.812
max_drawdown: -2.18% (2025-10-17)
win_rate: 100%
```

**乍看很美，但要诚实拆解**：
- 仅 7 笔已平仓事件，胜率 100% 是**样本太小**的偶然，不是 alpha
- 同期 A股 大盘整体上涨；策略大部分时间空仓，回撤小是「没怎么暴露」
- 平均算下来每月不到 2 笔交易，**几乎没吃成本**

### 3.3 完全随机 score 对照基准（同回测引擎、同阈值 0.51/0.49）

把 prediction 的 score 全部替换成 U(0,1) 随机数（相同种子可复现），跑同一个引擎：

```
n_trades: 981
total_return: -10.59%
annualized_return: -13.60%
sharpe: -0.827
max_drawdown: -18.79%
win_rate: 40%
profit_loss_ratio: 0.93
```

**对比结论**：
- 随机策略亏 10.6%，**主因是高频交易吃掉的成本**（佣金+印花+滑点 累计被收 980 次）
- 模型版赢的不是「预测准」，是「极少交易 → 极少吃成本 + 市场顺势」

### 3.4 总结一句话

回测引擎本身可用、A股 约束都生效、look-ahead 严控；
**模型预测力还不够，目前的「正收益」主要靠少交易和市场 β**，不是 alpha。
这与 P3b 模型 AUC≈0.5 的诚信弱基线完全一致 —— 不装水准高，留给后续 P5+ 迭代。

## 4. 用法

### 4.1 一条命令端到端

```python
from astock_quant.pipeline.run_direction import run_direction
r = run_direction(verbose=True)  # 默认 backtest 开
print(r["backtest_metrics"])
print(r["signals"].notes)
```

### 4.2 自定义回测参数

```python
from astock_quant.backtest.engine import BacktestConfig
from astock_quant.pipeline.run_direction import run_direction

cfg = BacktestConfig(
    initial_cash=1_000_000.0,
    buy_threshold=0.51, sell_threshold=0.49,
    max_positions=5,
    commission_rate=0.0003,
    stamp_tax_rate=0.0005,
    slippage_bps=5.0,
)
r = run_direction(backtest_config=cfg)
result = r["backtest"]  # BacktestResult 对象
result.equity_curve      # 净值曲线 DataFrame
result.trades            # 成交流水 DataFrame
result.positions         # 持仓时序快照 DataFrame
result.metrics           # 汇总指标 dict
```

### 4.3 独立调回测引擎（不跑训练）

```python
from astock_quant.backtest.engine import BacktestEngine
from astock_quant.data.dataset import build_price_panel

panel = build_price_panel()
predictions = [...]  # 你自己来源的 Prediction
res = BacktestEngine(panel).run(predictions)
```

### 4.4 独立产信号

```python
from astock_quant.signals.generator import SignalGenerator
sig = SignalGenerator(buy_threshold=0.55).generate(predictions)
sig.to_dataframe().to_csv("signals.csv", index=False)
```

### 8.1 `metrics.py` Sortino 边界 bug 兜底

**问题**：全空仓场景（0 笔交易、净值恒定）下 `excess = -daily_rf` 全部为负 → `downside_dev > 0` → 算出 `sortino ≈ -15.87`，与「没承担风险」直觉相违。Sharpe 已有 `std=0 → 0.0` 兜底，Sortino 漏掉。

**修复**：把判据统一到 `daily_returns.std() < 1e-12` —— 净值无波动时 Sharpe / Sortino 都直接返回 0。

```python
ret_std = float(daily_returns.std())
if ret_std < 1e-12:
    sharpe = 0.0
    sortino = 0.0  # ← 新增兜底，与 Sharpe 对齐
else:
    sharpe = ...
    sortino = ...
```

**测试守住**：`tests/test_backtest_engine.py::test_metrics_sortino_empty_positions`（净值恒定 20 日 → Sharpe / Sortino 都断言 0.0）。

### 8.2 `BacktestConfig` 重命名为 `BacktestRunConfig`，消除同名异字段歧义

**问题**：`config/settings.py:BacktestConfig`（项目级回测设置）与 `backtest/engine.py:BacktestConfig`（引擎运行参数）同名但字段集合完全不同 —— 读者容易混淆「这俩是同一个吗」。

**修复**：选 (a) 改名 ——
- `backtest/engine.py` 的 `BacktestConfig` → **`BacktestRunConfig`**（"单次回测运行的引擎参数"，语义更准）
- `config/settings.py` 的 `BacktestConfig` 保留不动（仍是"项目级回测设置"）
- 同步更新 `pipeline/run_direction.py`、`tests/test_backtest_engine.py` 的 import 和构造调用
- 新 docstring 写清两者的职责区分（项目配置 vs 单次运行覆盖）

**为什么没选 (b) 合并**：两者职责真的不同 —— 项目级配置稳定（起止日期、初始资金、费率），运行级参数频繁覆盖（阈值、max_positions、ST 名单）。合并会让一个类承担两种语义，反而更乱。
