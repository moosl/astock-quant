## 1. 建了什么

### 1.1 `contracts.py` —— 补全因子层 2 个契约（只动这 2 个）

| 契约 | 字段 | 说明 |
|---|---|---|
| `FactorValue` | factor_name, ticker, date, value, z_score?, percentile? | 单点查询用。debug 时取「ticker T 在日期 D 的 RSI」走这个。 |
| `FactorFrame` | data: DataFrame, factor_names: list[str] | 批量产出用。`data` 是 MultiIndex=(date, ticker)，columns=因子名，由 registry 拼出，模型层的输入特征 X。 |

`FactorFrame` 用 `model_config = ConfigDict(arbitrary_types_allowed=True)` 持有
DataFrame（Pydantic v2 写法），并暴露 `shape` / `n_observations` / `nan_ratio()` 辅助。

其它契约（Label / Prediction / BacktestResult / SignalReport）保持 stub 不动 ——
那是 model-engineer / backtest-engineer 的活。

### 1.2 `factors/base.py` —— BaseFactor ABC

抽象基类，所有因子继承。约束：
- `name: str` —— 因子标识（如 `momentum_20d`、`roe`、`main_inflow_5d`）
- `compute(panel, **kwargs) -> pd.Series` —— 输出 MultiIndex=(date, ticker) 的 Series

提供共享 helper：`_safe_div`（防 0/NaN 除）、`_winsorize`（截尾去极值）、
`_replace_inf`（inf→NaN）、`_group_by_ticker`（panel 按 ticker 分组）。
LLM 因子 stub 也继承同一个 BaseFactor —— 对下游 registry 无差别。

### 1.3 因子四件套（按 P1 蓝图）

| 文件 | 因子数 | 内容 |
|---|---:|---|
| `price_volume.py` | 13 | 动量 / 均值回归 / 波动率 / 量价 / 技术指标 |
| `fundamental.py` | 7 | 估值（PE/PB）/ 盈利（ROE/净利率/EPS）/ 成长（营收 yoy / 净利 yoy） |

## 3. 关键决策

### 3.1 防 look-ahead：因子层的纪律

P2 的 `cache.truncate_by_date` 是数据进 panel 的第一道闸；本层负责**不要在因子内
部破坏它**。具体做到三点：

1. **所有窗口操作（rolling / pct_change / shift）默认向后看**，pandas 这是默认
   行为，但要小心 —— 比如 `shift(-1)` 这种 forward-looking 一旦写出来就泄漏，
   本层代码全部禁用负数 shift。
2. **跨 ticker 的时序操作必须先 groupby ticker 再 transform**。例如：
   - `panel["close"].pct_change(20)` ❌ —— pct_change 会跨 ticker 串数据
   - `panel["close"].groupby(level="ticker").pct_change(20)` ✅
   ATR 因子最早写时用了 `.apply(_atr_one)`，导致索引重排，已改成 `groupby+transform`。
3. **「未来 N 日收益」这类必须丢给 labels/ 层**，因子层绝不允许出现 forward-looking
   窗口。这是阶段分工。

**端到端验证**：用 `curr_date=2024-06-30` 截断的 panel 跑因子，对比全量 panel 同
日期同因子值，**逐因子全样本对照** —— 25 / 25 因子在切点 bit-exact 一致（详见
5.4 节对照表）。**证明因子内部没有看未来。**

注：v1 实现里 `_winsorize` 用「整列后处理分位」做截尾边界，曾导致 3 个财务因子
（net_margin / revenue_growth_yoy / net_profit_growth_yoy）在切点出现数据依赖型
偏差（max abs diff 高达 0.47）—— auditor 全 25 因子复测发现后已修复（删因子层
全部 5 处 `_winsorize` 调用，下沉到 labels / preprocessing 层）。修复闭环和测试
固化见 5.4.1 / 5.4.2。

### 3.2 NaN 处理：分场景，不偷懒填 0

不同因子的 NaN 比例反映不同含义，因子层一律保留 NaN，让 model 层 / 标签层决定怎么处理：

| NaN 来源 | 处理 | 例子 |
|---|---|---|
| 数据起始处窗口没填满 | 自然 NaN | momentum_60d 前 60 行 |
| 数据源历史不够 | 自然 NaN | 资金流因子在 2025-11 前 |
| 财报里某字段缺失 | 自然 NaN | 老报告期没 EPS / ROE |
| 数学上不合理 | 强制 NaN | 负 PE / 负 PB（已实现） |
| inf / -inf | 替换 NaN | 除零导致的 inf（已实现） |

**不做的事**：因子层不做填充 / 不假设「缺数据 = 0」。0 在很多因子里是有意义值
（如净流入 0 = 当日零和），填 0 会造成假信号。

### 3.3 财务报告期 → 交易日的对齐

财务数据是季度粒度（按报告期发布），行情是日频。对齐用 forward-fill：
某交易日 T 的财务因子值 = 「报告期 <= T 的最近一期已发布报告」。

实现细节：
- `align_financials_to_panel(financials, panel, field)` —— 通用工具
- 每只票独立处理：先把 `[(report_period, value)]` 转 Series 按报告期排序、
  reindex 到该 ticker 的交易日，ffill
- panel 的 ticker level 可能是 int（CSV roundtrip 丢前导零，`000858` → `858`），
  用 `_build_ticker_map` 做 `lstrip("0")` 比对鲁棒匹配

**已知 TODO（财报发布滞后）**：严格说应该用「财报实际发布日」截断而不是「报告期
末日」—— 因为年报 3-4 月才发，即 2024-03 实际可见的最新报告还是 2023Q3。Stage 1
简化为「报告期末日 ≤ T 即可见」，会让每年 Q1（约 3 个月）的财务因子轻微 forward-
looking。已在 `fundamental.py` 模块 docstring 留 TODO，影响范围有限，但严格的因子
分析需修正（要求 `astock_source.get_financials` 同时返回 publish_date）。

### 3.4 PE / PB 历史几乎全 NaN —— 这是已知设计代价

P2 的 `astock_source.get_financials` 只把 PE / PB / 总股本 / 流通股本挂在「最新一期」
报告上（来自腾讯实时快照），历史报告期都是 None —— 这是**为了避免误用历史估值快照**
（你不能用 2026 的 PE 去对齐 2022 的交易日）。

结果：实测 PE / PB 在 31603 行中 **97.2% NaN**，只有最新约 60-100 行有值。
对 Stage 1 训练影响：训练集主要在历史区间，PE / PB 模型基本学不到。要让 PE / PB
真正可用需补齐历史 PE / PB 序列（要么用 akshare 取历史估值，要么从 BVPS / EPS 自己
反推） —— 这是后续 P3.1 的改进项，**Stage 1 知道这个代价，不强行处理**。

### 3.5 资金流 88.6% NaN —— P2 已警告，因子层鲁棒

`akshare stock_individual_fund_flow` 只覆盖最近约 6 个月，跟 4.4 年行情 panel
对齐时 240 / 2108 = 11.4% 有效覆盖。资金流因子全部用 `rolling(N, min_periods=1)`
—— 哪怕只有 1 天数据也能算 sum；窗口没覆盖到的日期保留 NaN（不填充）。
P2 早就警告过，本层照办。

### 3.6 stockstats vs 纯 pandas

P1/P2 都建议用 stockstats 算 MACD/BOLL/RSI/ATR。实测 stockstats 设计是「DataFrame
wrap 后当一只票算」—— 在多 ticker panel 上要按 ticker 逐个 wrap，反而比直接写公式
麻烦。所以本层选用纯 pandas：MACD 三层 `ewm` 嵌套总共 4 行；RSI 6 行；ATR True
Range 三分量 max 一行。stockstats 在依赖里保留作备选。

### 3.7 索引一致性

`registry.compute_factor_frame` 严格保证 **所有因子的输出索引等于 price_panel.index**：
- 计算前 panel 已是 sorted MultiIndex=(date, ticker)
- 单个因子返回的 Series 如果索引不一致（比如资金流因子来自 mf_panel），reindex 到
  price_panel.index
- 最终 `pd.concat(axis=1)` 拼成 FactorFrame.data，columns=因子名

实测验证：`ff.data.index.equals(price_panel.index) == True`，25 个因子全部对齐。

## 4. 怎么用因子层

### 4.1 一键产出 FactorFrame

```python
from astock_quant.data.dataset import prepare_stage1_data
from astock_quant.factors.registry import compute_factor_frame

data = prepare_stage1_data()  # P2 提供的一键入口
ff = compute_factor_frame(
    price_panel=data["prices"],
    moneyflow_panel=data["moneyflow"],
    financials=data["financials"],
)

ff.data           # DataFrame, MultiIndex=(date, ticker), 25 列因子
ff.factor_names   # ['momentum_5d', 'momentum_20d', ..., 'main_inflow_ratio']
ff.shape          # (31603, 25)
ff.nan_ratio()    # Series, 每个因子的 NaN 比例（debug 用）
```

### 4.2 逐日回测场景（防 look-ahead）

```python
from astock_quant.data.dataset import build_price_panel, build_moneyflow_panel, load_financials

curr = "2024-06-30"
price_p = build_price_panel(curr_date=curr)        # P2 在数据层截断
mf_p = build_moneyflow_panel(curr_date=curr)
fins = load_financials(curr_date=curr)

ff = compute_factor_frame(price_panel=price_p, moneyflow_panel=mf_p, financials=fins)
# ff 的 data 不会包含任何 > 2024-06-30 的行
```

### 4.3 自定义因子集（debug / 选股研究）

```python
from astock_quant.factors.price_volume import Momentum, RSI
from astock_quant.factors.registry import compute_factor_frame

ff = compute_factor_frame(
    price_panel=price_panel,
    factors=[Momentum(20), RSI(14)],  # 只跑这两个
```

#### 5.4.1 修复闭环：`_winsorize` 数据依赖型 look-ahead

Auditor 在初版（v1）发现 `_winsorize` 用「整列后处理分位」做截尾边界：

- 全量 panel：分位用 31603 个值算上界
- 截断 panel：分位用 18023 个值算上界
- 同一行（如 `date=2024-06-28, ticker=300059, factor=net_margin`）的原始 `s.quantile`
  不同 → 截断值被压到不同上界 → 在切点出现非零差异

具体差异（v1 实测）：

| 因子 | 非零差异行数 | max abs diff |
|---|---:|---:|
| net_margin | 359 | 0.317 |
| revenue_growth_yoy | 424 | 0.470 |
| net_profit_growth_yoy | 118 | 0.131 |

**修法（按 auditor 推荐方案 b）**：把因子层全部 5 处 `_winsorize` 调用删掉：

- `fundamental.py` PE._compute_inner（1.61 节）
- `fundamental.py` PB._compute_inner（1.79 节）
- `fundamental.py` NetMargin._compute_inner（2.21 节）
- `fundamental.py` `_yoy_growth_factor`（影响 RevenueGrowthYoY / NetProfitGrowthYoY）
- `moneyflow.py` MainInflowToAmount._compute_inner（1.73 节）

`base.py` 的 `_winsorize` **函数本体保留**（labels / preprocessing 层将来用），但
docstring 加了显式警告：「数据依赖型变换，不能在因子层使用 —— 必须用横截面或
expanding 分位，或下沉到 preprocessing 层」。PE 的 `s.where(s > 0)` 是合法 mask
（数学不合理直接 NaN，不依赖分位），保留。

**v2 复测结果**：全 25 因子在切点 bit-exact 一致（上方对照表）。

## 6. 给后续阶段的提醒

### 给 P4 labels 层 / models 层

1. **NaN 处理是下游的责任**。因子层故意不填充 / 不 dropna。labels 层可以选：
   - 训练前 `ff.data.dropna()` —— 简单但可能扔掉大量样本（资金流和 PE 拖累）
   - 训练前按列填充（`.fillna(0)` 或中位数）—— 留住样本但引入偏差
   - LightGBM 原生支持 NaN —— 推荐这个，无需填充

2. **建议 dropna 策略**：先按列选「NaN 比例 < 50% 的因子」作为 stable 集，
   单独看资金流 / PE 因子（NaN > 80%）的近期表现 —— 区分对待。

3. **横截面 vs 时序**：本层因子都是「每只票自己的时序」，没做横截面归一化（
   z-score across tickers in same date）。如果做 ranking 模型（③），需要在
   labels / preprocessing 层做横截面 rank/z-score。

### 给 P4 backtest 层

- **逐日回测必须用 `build_*_panel(curr_date=T)` + `compute_factor_frame`**，
  不要传全量 panel。已端到端验证截断生效，但纪律要刻在 pipeline 里。

### 给 P5 LLM 因子（Stage 2）

- `llm_factor.py` 已经是空 stub + Stage 2 实现路径 docstring。补实现时：
  1. 继承 `BaseFactor`，实现 `name` 和 `compute(panel, **kwargs)`
  2. 从 `kwargs.get("news")` 取 NewsItem list（需要 `compute_factor_frame` 加一个
     `news=...` 入参 —— 这是接入位 3 的轻微改动，本层已为它留了 `**kwargs` 透传）
  3. 注册到 `default_factors()` 的列表里
  4. 下游一行不用改

### 已知 TODO（Stage 1 不做，但要留意）

1. **财报发布日**：用报告期末日近似，年报场景轻微 forward-looking（见 3.3 节）
2. **PE/PB 历史**：97% NaN，要补齐需扩 P2 数据层
3. **横截面归一化**：本层只算原始因子，z-score / rank 留给 labels 层做（避免训练
   / 验证集泄漏）
